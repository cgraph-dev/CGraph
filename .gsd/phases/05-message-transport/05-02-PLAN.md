# Plan 05-02: Mobile Real-Time Message Transport

**Phase:** 05-message-transport **Wave:** 1 (parallel with 05-01) **Depends on:** —
**Requirements:** MSG-01, MSG-06, MSG-18, MSG-19

## Goal

Wire the mobile conversation screen's socket event handlers so that 1:1 messages deliver in
real-time with delivery receipt ACKs, read receipt display in message bubbles, auto-read on scroll
visibility, and privacy toggle integration — completing the end-to-end message transport on mobile.

## Context (from discovery)

**What already exists and works:**

- Mobile `SocketManager` (`apps/mobile/src/lib/socket.ts`, 1055 lines) — production-grade Phoenix
  Channel client with circuit breaker, session resumption (persists sessionId/lastSequence to
  SecureStore), exponential backoff + jitter, presence tracking, typing 5s auto-clear
- Mobile `chatStore` (`apps/mobile/src/stores/chatStore.ts`, 669 lines) — full CRUD, socket
  subscription (`subscribeToConversation` joins channel, handles new_message/updated/deleted/typing/
  reactions), 500-message cap, dedup via messageIds Set, snake_case normalization
- Conversation Screen (conversation-screen.tsx) with 20 hooks: `useConversationSocket`,
  `useSocketEventHandlers`, `usePresence`, `useMessageSending`, `useConversationData`, etc.
- `useSocketEventHandlers` already handles: `new_message`, `message_updated`, `message_deleted`,
  `message_pinned`, `message_read`, `reaction_added/removed`
- Message type already has `status?: 'sending' | 'sent' | 'delivered' | 'read' | 'failed'`,
  `delivered_at?`, `read_at?`, `is_optimistic?`
- WatermelonDB Message model has `status` column (sending/sent/delivered/read/failed)
- `settingsStore` has `showReadReceipts` and `showTypingIndicators` privacy toggles
- Offline queue supports `message`, `read_receipt` operation types
- Typing indicator component with 6 animation styles
- Backend broadcasts `msg_delivered` and `message_read` events on the conversation channel

**What's NOT wired (the gaps this plan closes):**

1. No delivery receipt ACK (`msg_ack`) pushed when receiving messages from other users — sender
   never transitions from "sent" to "delivered"
2. Message bubble doesn't render delivery status checkmarks from the `status` field
3. No auto-fire of `mark_read` when messages scroll into view — user must manually trigger
4. `showReadReceipts` / `showTypingIndicators` toggles exist but aren't gating actual behavior
5. `chatStore.sendMessage` waits for API response — no optimistic insert with `'sending'` status
6. `msg_delivered` and `message_read` socket events arrive via `useSocketEventHandlers` but don't
   update message `status` field in the chatStore

## Tasks

### Task 1: Wire delivery receipt ACK + status updates in chatStore

**Files:**

- `apps/mobile/src/stores/chatStore.ts`
- `apps/mobile/src/lib/socket.ts`

**Action:**

1. In `chatStore.ts`, add `updateMessageStatus` action:

   ```typescript
   updateMessageStatus: (conversationId: string, messageId: string, status: string) => void;
   ```

   Implementation finds the message in `messages[conversationId]` and updates its `status` field.

2. In `chatStore.ts`, update `subscribeToConversation` to:
   - On `new_message` from another user: after adding the message to the store, push `msg_ack` back
     through the channel to acknowledge delivery. The backend handles this at L215–L232 and
     broadcasts `msg_delivered` to the sender.
     ```typescript
     channel.push('msg_ack', { message_id: msg.id });
     ```
   - On `msg_delivered` event: call `updateMessageStatus(convId, msgId, 'delivered')`
   - On `message_read` event: call `updateMessageStatus(convId, msgId, 'read')`

3. In `chatStore.ts`, update `sendMessage` for optimistic insert:
   - Before the API call, add the message to the store with `status: 'sending'` and
     `is_optimistic: true`
   - On API success: update `status: 'sent'`, set `is_optimistic: false`, merge server-assigned
     fields (id, snowflake_id, timestamps)
   - On API failure: update `status: 'failed'` (user can retry)
   - Use `client_message_id` (already supported by backend) for idempotency

4. In `socket.ts`, ensure the `msg_delivered` and `message_read` events are passed through to the
   chatStore subscription callback. The socket already handles these events internally — verify they
   reach the chatStore's `subscribeToConversation` handler.

**Verify:** Send a message → shows "sending" → "sent" (API confirms) → "delivered" (when recipient
ACKs). Receive a message → `msg_ack` visible in WS traffic.

**Done:** Full delivery lifecycle tracked in chatStore. Optimistic sends show immediately.

### Task 2: Render message status + auto-read on scroll visibility

**Files:**

- `apps/mobile/src/screens/messages/conversation-screen/components/message-bubble.tsx`
- `apps/mobile/src/screens/messages/conversation-screen/hooks/useConversationSocket.ts`
- `apps/mobile/src/screens/messages/conversation-screen/hooks/useConversationData.ts`

**Action:**

1. In `message-bubble.tsx`, add delivery status indicator for sent messages:
   - Only show for messages where `senderId === currentUser.id`
   - Render based on `message.status`:
     - `'sending'` → clock icon (gray)
     - `'sent'` → single checkmark (gray) ✓
     - `'delivered'` → double checkmark (gray) ✓✓
     - `'read'` → double checkmark (blue/themed) ✓✓
     - `'failed'` → red exclamation with retry tap handler
   - Use Reanimated for smooth transitions between states
   - Position: bottom-right of bubble, next to timestamp

2. In `useConversationSocket.ts` or `useConversationData.ts`, implement auto-read on visibility:
   - Use `FlatList`'s `onViewableItemsChanged` callback (already available on inverted FlatList)
   - When incoming messages from the other user become visible (viewability threshold ≥ 50%):
     - Push `mark_read` through the channel with the latest visible message ID
     - Debounce to max once per 2 seconds to avoid flooding
   - Only auto-fire if:
     - The app is in the foreground (`AppState === 'active'`)
     - The conversation is the active conversation
     - `showReadReceipts` is enabled in settings

3. When opening a conversation that has unread messages, mark the latest visible message as read
   after a short delay (500ms) — this handles the case where the user opens a conversation and
   immediately sees new messages.

**Verify:** Open a conversation with new messages → read receipt fires after 500ms. Scroll through
messages → read receipts update as messages become visible. Sender sees checkmarks transition from
delivered → read.

**Done:** Message bubbles show real delivery status. Auto-read fires on message visibility.

### Task 3: Privacy toggle integration + typing refinements

**Files:**

- `apps/mobile/src/screens/messages/conversation-screen/hooks/useMessageSending.ts`
- `apps/mobile/src/screens/messages/conversation-screen/hooks/useSocketEventHandlers.ts`
- `apps/mobile/src/stores/chatStore.ts` (typing refinement)
- `apps/mobile/src/screens/messages/conversation-screen/conversation-screen.tsx`

**Action:**

1. Wire `showReadReceipts` toggle (from `settingsStore`):
   - If `showReadReceipts === false`:
     - Do NOT push `mark_read` events through the channel (sender's messages stay at "delivered")
     - Do NOT fire auto-read on scroll visibility
     - Do NOT render blue checkmarks on incoming messages (stay gray double-check)
   - If `showReadReceipts === true` (default): full read receipt behavior

2. Wire `showTypingIndicators` toggle:
   - If `showTypingIndicators === false`:
     - Do NOT push typing events through the channel (contacts don't see typing)
     - Do NOT render the typing indicator component in the conversation view
   - If `showTypingIndicators === true` (default): full typing indicator behavior

3. In `useMessageSending.ts`, add typing debounce:
   - Current: every keystroke may trigger a typing event
   - Fix: throttle `sendTyping(true)` to max once per 3 seconds
   - After 5 seconds of no keystroke activity, auto-send `sendTyping(false)`
   - Clear the stop-typing timer on new keystrokes

4. In `useSocketEventHandlers.ts`, add typing auto-clear safety net:
   - When receiving a `typing` event with `is_typing: true`, start a 6s timer
   - If no follow-up event arrives within 6s, auto-clear the typing state for that user
   - This prevents "stuck typing" if the stop event is lost

5. In `conversation-screen.tsx`, conditionally render the typing indicator:
   - Only show if `showTypingIndicators` is enabled
   - Pass the conversation's `typingUsers` from chatStore, filtered to exclude current user

**Verify:**

- Disable read receipts in settings → partner's messages stay at "delivered" (gray checkmarks)
- Re-enable → existing unread messages get marked as read on next open
- Disable typing indicators → partner doesn't see typing, user doesn't see partner typing
- Type rapidly → only 1 typing event per 3s in WS traffic
- Stop typing → indicator clears within 6s even if stop event dropped

**Done:** Privacy toggles gate all receipt/typing behavior. Typing is debounced with auto-clear.
