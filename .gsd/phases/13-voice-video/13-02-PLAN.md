# Plan 13-02: LiveKit SFU Integration — Group Calls

## Context

The current call system uses peer-to-peer mesh topology (each peer connects to every other peer). This works for 1:1 calls but breaks at 3+ participants (exponential bandwidth). LiveKit is an open-source SFU (Selective Forwarding Unit) that routes media through a server.

**What exists:**
- Backend `webrtc.ex` (451L) GenServer manages room lifecycle. `signaling.ex` has `sfu_enabled?()` flag and `sfu_url` config placeholder — but no actual LiveKit integration.
- Config `runtime.exs` has `WEBRTC_SFU_ENABLED` and `WEBRTC_SFU_URL` env vars.
- Web has `video-grid.tsx` (151L) that renders multiple video tiles — can be adapted for LiveKit tracks.
- Web has comprehensive type definitions for group calls.

**What's needed:**
1. LiveKit server SDK for backend (JWT token generation, room management)
2. `livekit-client` SDK for web
3. Backend controller to issue LiveKit tokens
4. Web LiveKit service + React hooks
5. Local dev environment (Docker Compose)
6. Conditional routing: P2P for 1:1, LiveKit for group

## Tasks

### Task 1: Backend LiveKit SDK integration + config

**Files:** `apps/backend/mix.exs`, `apps/backend/config/runtime.exs`, `apps/backend/lib/cgraph/webrtc/livekit_token.ex`

**Actions:**
1. Add `livekit_server_sdk` hex dependency (or implement JWT token generation directly — LiveKit tokens are standard JWTs with specific claims)
2. Update runtime.exs config:
   ```elixir
   config :cgraph, CGraph.WebRTC.LiveKit,
     api_key: env("LIVEKIT_API_KEY"),
     api_secret: env("LIVEKIT_API_SECRET"),
     url: env("LIVEKIT_URL") || "ws://localhost:7880"
   ```
3. Create `livekit_token.ex`:
   - `generate_token(room_name, participant_identity, opts)` — builds JWT with:
     - `iss` = api_key, `sub` = identity, `video` = grant object
     - Grant: `roomJoin: true, room: room_name, canPublish: true, canSubscribe: true`
   - Sign with HMAC-SHA256 using api_secret
   - TTL: 6 hours default

### Task 2: Backend LiveKit room management module

**Files:** `apps/backend/lib/cgraph/webrtc/livekit.ex`

**Actions:**
1. Create `CGraph.WebRTC.LiveKit` module:
   - `create_room(name, opts)` — POST to LiveKit server API `/twirp/livekit.RoomService/CreateRoom`
   - `list_rooms()` — list active rooms
   - `delete_room(name)` — close a room
   - `list_participants(room_name)` — get active participants
   - `remove_participant(room_name, identity)` — kick from room
   - `mute_participant(room_name, identity, track_sid)` — server-side mute
2. Use Protobuf-over-HTTP (Twirp) for LiveKit server API
3. Room naming convention: `group_{group_id}_channel_{channel_id}` for deterministic mapping

### Task 3: Backend LiveKit token controller

**Files:** `apps/backend/lib/cgraph_web/controllers/api/v1/livekit_controller.ex`, `apps/backend/lib/cgraph_web/router.ex`

**Actions:**
1. Create `LiveKitController`:
   - `create_token/2` — POST /api/v1/livekit/token
     - Params: `room_name`, `channel_id` (optional)
     - Auth: requires authenticated user
     - Authorization: if channel_id provided, verify user is member of the group owning that channel and has CONNECT permission
     - Returns: `{token: "jwt...", url: "wss://livekit-server"}`
2. Add route: `post "/livekit/token", LiveKitController, :create_token`
3. Integrate with existing room lifecycle:
   - When token is issued for a room that doesn't exist yet, create it in the backend registry + LiveKit

### Task 4: Hybrid routing in webrtc.ex (P2P vs SFU)

**Files:** `apps/backend/lib/cgraph/webrtc/webrtc.ex`

**Actions:**
1. Modify `create_room/2` to accept `:mode` option:
   - `:p2p` (default for 1:1 calls) — existing ETS room + Phoenix Channel signaling
   - `:sfu` (for group calls) — create LiveKit room + backend registry entry
2. Add `escalate_to_sfu/1` — when a P2P call gets a 3rd participant, migrate to SFU:
   - Create LiveKit room, issue tokens to existing participants, signal clients to reconnect
3. Modify `join_room/3` to check room mode and either continue P2P or redirect to LiveKit

### Task 5: Web LiveKit client integration

**Files:** `apps/web/package.json`, `apps/web/src/lib/webrtc/livekitService.ts`

**Actions:**
1. Add `livekit-client` dependency
2. Create `livekitService.ts`:
   - `connect(url, token)` — creates `Room` instance, connects, returns room handle
   - `publishLocalTracks(room, opts)` — publish audio + video tracks
   - `disconnect(room)` — leave and cleanup
   - Event handling: `RoomEvent.TrackSubscribed`, `TrackUnsubscribed`, `ParticipantConnected`, `ParticipantDisconnected`, `ActiveSpeakersChanged`
   - Adaptive quality: `room.setSubscribedTrackQuality(track, quality)`
3. Export: `LiveKitService` singleton with connect/disconnect/publish/setQuality

### Task 6: Web LiveKit React hooks + group call UI

**Files:** `apps/web/src/modules/calls/hooks/useLiveKitRoom.ts`, `apps/web/src/modules/calls/components/group-call-view.tsx`, `apps/web/src/modules/calls/components/livekit-participant-tile.tsx`

**Actions:**
1. Create `useLiveKitRoom` hook:
   - Fetch token from POST /api/v1/livekit/token
   - Connect to LiveKit room via `livekitService.connect()`
   - State: `participants`, `localTracks`, `activeSpeakers`, `connectionState`, `isMuted`, `isVideoOn`
   - Actions: `toggleMute`, `toggleVideo`, `startScreenShare`, `stopScreenShare`, `leaveRoom`
   - Cleanup on unmount
2. Create `group-call-view.tsx`:
   - Uses `useLiveKitRoom` hook
   - Renders participant grid (2×2, 3×3, speaker view)
   - Active speaker highlight
   - Control bar: mute, video, screen share, leave
   - Participant count badge
3. Create `livekit-participant-tile.tsx`:
   - Renders single participant video/audio
   - Shows name, mute indicator, speaking indicator
   - Handles video on/off gracefully (show avatar when video off)

### Task 7: Docker Compose for local LiveKit dev server

**Files:** `infrastructure/docker/docker-compose.livekit.yml`

**Actions:**
1. Create Docker Compose with LiveKit server:
   ```yaml
   services:
     livekit:
       image: livekit/livekit-server:latest
       ports:
         - "7880:7880"   # HTTP
         - "7881:7881"   # WebRTC TCP
         - "50000-50100:50000-50100/udp"  # WebRTC UDP
       environment:
         LIVEKIT_KEYS: "devkey: secret"
       command: --dev --bind 0.0.0.0
   ```
2. Add development env vars to backend config:
   ```
   LIVEKIT_API_KEY=devkey
   LIVEKIT_API_SECRET=secret
   LIVEKIT_URL=ws://localhost:7880
   ```
