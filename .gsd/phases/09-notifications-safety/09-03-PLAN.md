# Plan 09-03: Notification Wiring Audit & Gap Fixes

## Context

Discovery revealed that NOTIF-01 (mobile push), NOTIF-02 (web push), NOTIF-03 (notification center), NOTIF-08 (email digest), and AUTH-08 (account deletion + data export) are **fully implemented** at the individual component level. However, we need to verify:

1. **E2E wiring** — do new messages actually trigger push notifications for offline users?
2. **Integration with Plan 09-01/09-02** — do muted conversations and DND suppress delivery correctly?
3. **Minor gaps** — data export UI exposure, lapsed engagement targeting, shared types consolidation

This is a verification + polish plan, not a major feature build.

## Tasks

### E2E Push Wiring Audit

1. **Trace message → push notification chain**
   - Verify: `conversation_channel.ex` `handle_in("new_message")` → `Notifications.notify/4` or similar → `notification_worker.ex` → `push_service.ex` → platform sender
   - If any link is missing, wire it
   - Check: offline users (not connected to channel) get push notification
   - Check: online users (connected to channel) do NOT get redundant push

2. **Trace message → web push chain**
   - Same as above but for web push path
   - Verify service worker receives and displays notification
   - Verify clicking notification opens correct conversation

3. **Trace message → notification center entry**
   - Verify: new messages create a `Notification` record in DB
   - Verify: notification center shows it with correct type, grouping, and action_url
   - Verify: real-time delivery via `notification_channel` or user socket

### Integration with Per-Target Preferences (09-01)

4. **Wire preference check into delivery pipeline**
   - After 09-01 is complete, verify that `delivery.ex` checks `Preferences.should_deliver?` before:
     - Sending push notification
     - Sending web push notification
     - Creating notification center entry (or mark it as muted/low-priority)
   - Write integration test: mute conversation → send message → no push, no web push

### Integration with DND (09-02)

5. **Wire DND check into delivery pipeline**
   - After 09-02 is complete, verify that `should_notify?` combined with `in_quiet_hours?` and `dnd_until` suppresses all delivery
   - Write integration test: enable DND → send message → no notifications of any kind

### Email Digest Lapsed Engagement

6. **Add `last_seen_at` based targeting**
   - Check if `users` table has a `last_seen_at` column (likely yes — it's a common field)
   - If not, add migration
   - Update `email_digest_worker.ex` to filter: only send digest to users where `last_seen_at < N days ago`
   - This prevents sending digests to daily active users (they don't need re-engagement)

7. **Verify digest email templates**
   - Check that email template renders correctly with unread counts, trending content
   - Ensure unsubscribe link is present (GDPR/CAN-SPAM requirement)

### Data Export UI

8. **Web: Add data export section to settings**
   - Check if `pages/settings/` already has a data export page or section
   - If not, create a simple section with:
     - "Request Data Export" button → calls `POST /api/v1/users/me/export`
     - Status display: "Export in progress..." / "Download ready" with link
     - Link to download when ready
   - If already exists, verify wiring

9. **Mobile: Add data export to account settings**
   - Check if `account-screen.tsx` already has export option
   - If not, add "Download My Data" row with same flow as web
   - If already exists, verify wiring

### Shared Types Consolidation

10. **Extend `packages/shared-types/src/notifications.ts`**
    - Add types that are currently defined independently in web and mobile:
      ```typescript
      export interface Notification {
        id: string;
        type: 'message' | 'social' | 'group' | 'forum' | 'system';
        title: string;
        body: string;
        actionUrl?: string;
        groupKey?: string;
        count?: number;
        readAt?: string;
        clickedAt?: string;
        createdAt: string;
      }

      export interface PushPayload {
        title: string;
        body: string;
        data?: Record<string, string>;
        badge?: number;
        sound?: string;
      }

      export interface NotificationStats {
        total: number;
        unread: number;
      }
      ```
    - Update web `notificationStore.impl.ts` and mobile `notificationStore.ts` to import from shared-types

### Integration Tests

11. **Write backend integration tests**
    - Test: message sent → notification created for recipient (not sender)
    - Test: muted conversation → no notification created
    - Test: DND active → no notification delivered
    - Test: mentions-only mode → regular message = no notification, @mention = notification
    - Test: data export request → Oban job created → export file generated

## Verification

- Send a message while recipient is offline → verify push notification received on mobile
- Send a message while recipient has web push enabled → verify browser notification
- Open notification center → verify message notification appears with correct info
- Click notification → verify navigation to correct conversation
- Request data export → download file → verify contents include messages, profile, settings
