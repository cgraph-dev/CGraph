# v0.7.44 Release Notes - Secure Random UID System

**Release Date:** January 11, 2026

## Overview

This release replaces the sequential user ID system with a cryptographically secure random 10-digit
UID to prevent security vulnerabilities associated with enumerable user identifiers.

## Security Improvements

### Problem with Sequential IDs

The previous system used sequential integers (e.g., `#0001`, `#0002`, `#0003`) which exposed several
security risks:

| Vulnerability                     | Description                                         |
| --------------------------------- | --------------------------------------------------- |
| **User Enumeration**              | Attackers could easily guess valid user IDs         |
| **Business Intelligence Leakage** | Competitors could estimate total user count         |
| **Targeted Attacks**              | Easy to iterate through all users for scraping/spam |
| **Account Age Exposure**          | Lower numbers revealed early adopters               |

### New Random UID System

- **Format:** 10 random digits (e.g., `#4829173650`)
- **Generation:** Cryptographically random, non-sequential
- **Uniqueness:** Guaranteed unique via database constraint + retry loop
- **Capacity:** 9 billion possible UIDs (10-digit numbers starting with 1-9)

## Technical Changes

### Database Migration

- Added new `uid` column (VARCHAR 10) to users table
- Created PostgreSQL function `generate_random_uid()` for unique UID generation
- Created trigger `users_set_uid_trigger` for automatic UID assignment on user creation
- Added unique index on `uid` column
- Legacy `user_id` (integer) retained for backward compatibility

### Backend Changes

- **User Schema** (`accounts/user.ex`): Added `uid` field with `read_after_writes: true`
- **format_user_id/1**: Now returns `#<uid>` (10-digit) with fallback to legacy format
- **get_user_by_user_id/1**: Searches by `uid` first, falls back to legacy `user_id`
- **search_users/2**: Updated to search by 10-digit UID or legacy 1-4 digit ID

### API Response Changes

All user API responses now include:

```json
{
  "id": "uuid-here",
  "uid": "4829173650",
  "user_id": 1,
  "user_id_display": "#4829173650",
  ...
}
```

### Frontend Changes

- **Web authStore.ts**: Added `uid` field to User interface
- **Web friendStore.ts**: Updated UID detection regex for 1-10 digit UIDs
- **Mobile types/index.ts**: Added `uid` field to User interface
- **Mobile AddFriendScreen.tsx**: Updated examples to show 10-digit UID format

## Migration Path

### For Existing Users

- Migration automatically generates random 10-digit UIDs for all existing users
- Legacy `user_id` (integer) is preserved for backward compatibility
- Old sequential IDs still work when searching (falls back if UID not found)

### For New Users

- Automatically assigned a random 10-digit UID on account creation
- No action required from developers or users

## Breaking Changes

None. The system maintains full backward compatibility:

- Legacy `#0001` format still works for existing users
- API responses include both `uid` and `user_id` fields
- Search functions check both fields
