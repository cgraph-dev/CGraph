# v0.7.17 Critical Production Fixes

**Release Date:** 2026-01-08  
**Status:** Emergency Release  
**Severity:** CRITICAL

## Executive Summary

v0.7.17 resolves three critical production issues discovered during v0.7.16 testing:

1. Backend crash on message history load (Protocol.UndefinedError)
2. Persistent presence join/leave loop (100-200 events/sec)
3. Voice message 500 error (package version mismatch)

All issues are now resolved with proper architectural fixes.

## Issue 1: Backend Message History Crash

### Symptoms

```
[error] GenServer #PID<0.1234.0> terminating
** (Protocol.UndefinedError) protocol Enumerable not implemented for {[...], %{...}} of type Tuple
    (elixir 1.19.4) lib/enum.ex:1: Enumerable.impl_for!/1
    (elixir 1.19.4) lib/enum.ex:166: Enumerable.reduce/3
    (elixir 1.19.4) lib/enum.ex:4507: Enum.map/2
```

### Root Cause

**File:** `apps/backend/lib/cgraph_web/channels/conversation_channel.ex:61`

```elixir
# WRONG - treats tuple as list
def handle_info(:after_join, socket) do
  conversation = socket.assigns.conversation
  messages = Messaging.list_messages(conversation, limit: 50)  # Returns {messages, meta}
  serialized_messages = Enum.map(messages, &MessageJSON.message_data/1)  # CRASH!
  ...
end
```

**File:** `apps/backend/lib/cgraph/messaging.ex:238`

```elixir
def list_messages(conversation, opts \\ []) do
  # ... query building ...
  {messages, meta}  # Returns TUPLE, not list
end
```

### Solution

Proper tuple unpacking with Elixir pattern matching:

```elixir
def handle_info(:after_join, socket) do
  conversation = socket.assigns.conversation
  {messages, _meta} = Messaging.list_messages(conversation, per_page: 50)
  serialized_messages = Enum.map(messages, &MessageJSON.message_data/1)
  ...
end
```

### Impact

- ✅ Message history loads without crashing
- ✅ Pagination metadata properly handled
- ✅ Consistent return type handling across codebase

## Issue 2: Persistent Presence Loop

### Symptoms

```
LOG [Socket] User cc9e2c4c joined conversation:2363d791
LOG [Socket] User cc9e2c4c left conversation:2363d791
LOG [Socket] User cc9e2c4c joined conversation:2363d791
LOG [Socket] User cc9e2c4c left conversation:2363d791
(repeats 100-200 times per second)
```

### Root Cause Analysis

**Why v0.7.16 Failed:**

v0.7.16 implemented socket-level debouncing which works correctly, but it cannot prevent
component-level issues:

1. **React Navigation Unmounts Components**
   - Navigating between screens unmounts/remounts components
   - All refs (including `hasInitializedRef`) reset to initial values
   - v0.7.16's "one-time initialization" pattern defeated

2. **useEffect Cleanup Pattern**
   - v0.7.16 called `leaveChannel()` in useEffect cleanup
   - Component remount triggers immediate `joinChannel()`
   - Creates join → unmount → leave → mount → join loop

3. **Dependency Array**
   - `[conversationId]` dependency necessary for conversation switching
   - But triggers re-execution on ANY state change propagating through deps

**File:** `apps/mobile/src/screens/messages/ConversationScreen.tsx` (v0.7.16)

```typescript
// FLAWED v0.7.16 approach
const hasInitializedRef = useRef(false);

useEffect(() => {
  if (hasInitializedRef.current) return; // ❌ Ref resets on remount!
  hasInitializedRef.current = true;

  socketManager.joinChannel(channelTopic);

  return () => {
    // ❌ Calling leaveChannel causes immediate rejoin on remount!
    socketManager.leaveChannel(channelTopic);
  };
}, [conversationId]);
```

### Solution

**Strategy:** Keep channels alive across navigation, rely on socket manager debouncing.

```typescript
// ✅ CORRECT v0.7.17 approach
useEffect(() => {
  isMountedRef.current = true;
  const channelTopic = `conversation:${conversationId}`;

  // joinChannel has built-in debouncing - safe to call on every mount
  socketManager.joinChannel(channelTopic);

  const unsubscribe = socketManager.onChannelMessage(channelTopic, ...);
  cleanupRef.current = unsubscribe;

  return () => {
    isMountedRef.current = false;
    if (cleanupRef.current) {
      cleanupRef.current();  // Remove listeners only
      cleanupRef.current = null;
    }
    // ✅ Don't leave channel - socket manager keeps it alive for session
    // ✅ The join debouncing prevents duplicate joins on remount
  };
}, [conversationId]);
```

**Socket Manager Debouncing** (unchanged from v0.7.16):

```typescript
joinChannel(topic: string): Channel | null {
  const now = Date.now();
  const lastAttempt = this.lastJoinAttempts.get(topic) || 0;

  if (now - lastAttempt < 1000) {  // 1-second debounce
    return this.channels.get(topic) || null;
  }

  this.lastJoinAttempts.set(topic, now);
  // ... proceed with join ...
}
```

### Impact

- ✅ Join/leave events reduced from 100-200/sec to 1-2 events on navigation
- ✅ Channels persist across component remounts
- ✅ Debouncing prevents duplicate joins
- ✅ Clean message listener management with unmount cleanup

## Issue 3: Voice Message 500 Error

### Symptoms

```
ERROR Error sending voice message: [AxiosError: Request failed with status code 500]
WARN expo-av@15.0.2 - expected version: ~16.0.8
```

### Root Cause

**File:** `apps/mobile/package.json`

```json
{
  "dependencies": {
    "expo-av": "~15.0.2" // ❌ Incompatible with Expo SDK 54

}
```

### Solution

Update to SDK-compatible version:

```json
{
  "dependencies": {
    "expo-av": "~16.0.8" // ✅ Compatible with Expo SDK 54
  }
}
```

**Command:**

```bash
pnpm add expo-av@16.0.8
```

### Impact

- ✅ Voice message recording works correctly
- ✅ No SDK version warnings
- ✅ Audio playback compatible with SDK 54

## Architecture Lessons

### Pattern: When to Leave Channels

**❌ DON'T leave channels when:**

- Component unmounts but will likely remount (navigation)
- User is still in the conversation context
- Channel has short-lived component subscribers

**✅ DO leave channels when:**

- User logs out (session end)
- WebSocket disconnects
- Explicit user action (leave group, close conversation)

### Pattern: Component Lifecycle vs Session Lifecycle

**Component Lifecycle:**

- React Navigation unmounts/remounts frequently
- Refs reset on unmount
- State recreated on mount
- Best for: UI state, temporary subscriptions

**Session Lifecycle:**

- Singleton managers persist across navigation
- State maintained in external singletons (socketManager)
- Cleanup only on session end (logout, disconnect)
- Best for: WebSocket connections, channel subscriptions, auth state

### Pattern: Tuple Returns in Elixir

When functions return tuples (common for pagination):

```elixir
# ✅ ALWAYS unpack with pattern matching
{data, metadata} = MyContext.list_items(opts)

# ❌ NEVER treat tuple as list
items = MyContext.list_items(opts)  # Returns tuple!
Enum.map(items, &transform/1)  # CRASH!
```
