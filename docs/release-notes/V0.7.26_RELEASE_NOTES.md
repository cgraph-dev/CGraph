# CGraph v0.7.26 Release Notes

**Release Date:** January 5, 2026  
**Status:** Production-Ready  
**Breaking Changes:** Yes (see Migration section)

---

## What's in This Release

This is the release where CGraph becomes genuinely production-ready. We've been heads-down fixing the kind of issues that only surface when you actually try to deploy to real infrastructure—missing config files, security gaps, and a few embarrassing bugs that made it past our test suite.

The short version: if you tried deploying CGraph before this release and hit weird Docker failures or your encryption wasn't working right, those problems are fixed now.

We've also added the legal documents (Privacy Policy, Terms of Service) that app stores require, built out a proper content moderation system, and added the E2EE key verification UI that was conspicuously missing.

---

## The Big Fixes

### Production Config Was Missing

Okay, this one's kind of embarrassing. We had development config (`dev.exs`) and test config (`test.exs`), but `prod.exs` literally didn't exist. Docker builds failed, Fly.io deploys failed, basically anything production-related broke immediately.

**Fixed:** There's now a proper [config/prod.exs](apps/backend/config/prod.exs) with:
- Secure cookie settings (httponly, secure, SameSite=Strict)
- JSON logging that doesn't leak credentials  
- Oban job cleanup (keeps 6 hours of history)
- Rate limiting presets for different endpoint types

### Encryption Keys Were Being Generated On-the-Fly

This was bad. If you forgot to set `ENCRYPTION_KEY` in production, the app would silently generate a random key at startup. Which sounds fine until you realize: restart the app, new key, all your encrypted data is now unreadable.

**Fixed:** In production, missing `ENCRYPTION_KEY` now crashes the app immediately with a clear error message. You'll know exactly what's wrong instead of discovering data loss later.

```elixir
# Now in production
defp fallback_key do
  if Mix.env() == :prod do
    raise "ENCRYPTION_KEY environment variable is required in production!"
  end
  :crypto.strong_rand_bytes(32)  # Only allowed in dev/test
end
```

### Group Conversations Returned 500 Errors

Trying to create a group conversation (multi-user DM) returned a 500 server error because the feature isn't implemented yet. That's confusing for API consumers—is the server broken? Is my request malformed?

**Fixed:** Now returns a 400 with a clear message: "Group conversations are planned for v0.8.0. Use Groups for multi-user communication."

**Solution:** Returns 400 with clear message directing to roadmap.

```json
{
  "error": "Group conversations are not yet supported",
  "message": "Group messaging is planned for v0.8.0. Use Groups feature for multi-user communication.",
  "docs": "https://docs.cgraph.io/roadmap"
}
```

## Security Improvements

### 1. CSRF Protection Enhanced

Changed cookie `SameSite` from `Lax` to `Strict` for maximum CSRF protection.

### 2. Required Environment Variables

Production now requires:
- `SECRET_KEY_BASE` (no default)
- `DATABASE_URL` (no default)  
- `ENCRYPTION_KEY` (no default)
- `GUARDIAN_SECRET_KEY` (no default)

## Migration Guide

### 1. Environment Variables (Required)

Add these to your production environment:

```bash
# Generate with: mix phx.gen.secret
export SECRET_KEY_BASE="your-64-char-secret"

# Generate with: openssl rand -base64 32
export ENCRYPTION_KEY="your-32-byte-base64-key"

# Generate with: mix phx.gen.secret
export GUARDIAN_SECRET_KEY="your-jwt-signing-secret"

# PostgreSQL connection
export DATABASE_URL="postgres://user:pass@host:5432/cgraph_prod"
```

### 2. Database Migration

Run the new moderation tables migration:

```bash
mix ecto.migrate
```

### 3. Docker Compose

Update your `.env` file - the following no longer have defaults:
- `SECRET_KEY_BASE`
- `DATABASE_URL`
- `ENCRYPTION_KEY`
- `GUARDIAN_SECRET_KEY`

## Breaking Changes

1. **Missing ENCRYPTION_KEY now fails startup** (previously silently used ephemeral key)
2. **docker-compose.yml no longer has default secrets** (must provide via .env)
3. **Cookie SameSite changed to Strict** (may affect cross-site OAuth flows)
