<objective>
Wire PQ-bridge into mobile E2EE flow and auto-bootstrap on login.

Purpose: Mobile has a complete PQXDH + Triple Ratchet bridge (`pq-bridge.ts`) that wraps
`@cgraph/crypto`, but it's not connected to the message flow. The e2eeStore still uses the legacy
X3DH implementation. This plan switches the mobile crypto path to PQXDH and adds auto-bootstrap â€”
satisfying E2EE-01, E2EE-08 (already done via SecureStore), and E2EE-09 for mobile.

Output: Mobile e2eeStore encrypts/decrypts via pq-bridge, auto-bootstraps keys on login, uploads KEM
prekeys. </objective>

  <name>Task 1: Switch e2eeStore encrypt/decrypt to use pq-bridge</name>
  <files>
    apps/mobile/src/lib/crypto/store/e2eeStore.ts
    apps/mobile/src/lib/crypto/pq-bridge.ts

  <action>
    1. In `e2eeStore.ts`, replace the `encryptMessage` implementation to delegate to `pq-bridge.ts`:
       - Import functions from `pq-bridge.ts` (`initiateSession`, `encryptMessage` as `pqEncrypt`, `decryptMessage` as `pqDecrypt`, `generateKeyBundle`)
       - In `encryptMessage(recipientId, plaintext)`:
         a. Check if a PQ session exists for this recipient (check pq-bridge session store)
         b. If no session: fetch recipient's prekey bundle via API (`/api/v1/e2ee/keys/:user_id`), call `pq-bridge.initiateSession(bundle, store)` to create a PQXDH session
         c. Call `pq-bridge.encryptMessage(recipientId, plaintext)` â†’ returns `{ ciphertext, header, protocol_version }`
         d. Return encrypted payload with `is_encrypted: true`, protocol version `pqxdh_v1`

    2. Similarly update `decryptMessage(senderId, encryptedPayload)`:
       - Check `protocol_version` field in the payload
       - For `pqxdh_v1`: delegate to `pq-bridge.decryptMessage(senderId, encryptedPayload)`
       - For legacy `x3dh`/`classical_v1`/`classical_v2`: keep existing `e2ee.ts` decrypt path for backward compat
       - Return plaintext string

    3. In `setupE2EE` action, switch from `e2ee.ts` key generation to `pq-bridge.generateKeyBundle()`:
       - Generate identity key pair + signed prekey + 100 OTKs + KEM prekey (ML-KEM-768)
       - Register full bundle via POST /api/v1/e2ee/keys (include kyber_prekey, kyber_prekey_id, kyber_prekey_signature)
       - Store private keys in SecureStore with `WHEN_UNLOCKED_THIS_DEVICE_ONLY` (pq-bridge already does this)

    4. Use the existing `setupPromiseRef` pattern from `useE2EE.ts` to prevent concurrent session initialization race conditions.

    IMPORTANT: Keep backward compat â€” messages from X3DH sessions must still decrypt. Do NOT remove `e2ee.ts` imports.

  </action>

       - After `checkStatus()` completes, if the result indicates E2EE is NOT set up (no identity key found), automatically call `setupE2EE()` from the e2eeStore.
       - Pattern:
         ```
         const status = await e2eeStore.checkStatus();
         if (!status.isSetUp && authState.isAuthenticated) {
           await e2eeStore.setupE2EE();
         ```

       - This runs on provider mount (after login), making E2EE bootstrap invisible to the user.

    2. Handle the loading state: while `setupE2EE` runs, the store's `isInitializing` flag should be `true`. The E2EE provider should expose this state so UI can optionally show "Setting up encryption..." if it takes > 500ms.

    3. Handle errors gracefully: if `setupE2EE` fails (network error, etc.), log the error and set `setupError` in the store. Do NOT crash the app â€” messaging should still work but will be unencrypted until next attempt. The provider should retry on next app foreground.

    4. Ensure `checkStatus` in the e2eeStore checks for BOTH legacy (X3DH) and new (PQXDH) key presence. If legacy keys exist but no KEM prekey, trigger a KEM prekey upload to upgrade the bundle.

    IMPORTANT: The auto-bootstrap must not block the app render. Use a background task pattern â€” the user should see the chat UI immediately while keys are generated in the background.
