<objective>
Build multi-device E2EE key sync backend — cross-signing, trust chain, and sync endpoints.

Purpose: This is the only genuinely new feature in Phase 7. When a user adds a new device, it needs
its own identity key, and existing devices must cross-sign new device keys to establish a trust
chain. The backend needs a cross-signature storage schema, a sync endpoint for encrypted key
material, and a device trust verification API. Satisfies E2EE-04 backend portion.

Output: Backend schema, context modules, and API endpoints for multi-device key sync with
cross-signing. </objective>

  <name>Task 1: Create cross-signing schema, migration, and context</name>
  <files>
    apps/backend/priv/repo/migrations/*_add_e2ee_cross_signatures.exs
    apps/backend/lib/cgraph/crypto/e2ee/cross_signing.ex
  </files>
  <action>
    1. Create a migration for the `e2ee_cross_signatures` table:
       - `id` (UUID, primary key)
       - `signer_device_id` (UUID, FK to e2ee_identity_keys — the device doing the signing)
       - `signed_device_id` (UUID, FK to e2ee_identity_keys — the device being signed)
       - `user_id` (UUID, FK to users — both devices must belong to same user)
       - `signature` (binary — the actual cryptographic signature)
       - `algorithm` (string — e.g., "ed25519" or "ecdsa-p256")
       - `status` (string — "pending" | "verified" | "revoked")
       - `inserted_at`, `updated_at` timestamps
       - Unique constraint: `[:signer_device_id, :signed_device_id]`
       - Index on `user_id`

    2. Create `cross_signing.ex` context module:
       - `create_cross_signature(signer_device_id, signed_device_id, user_id, signature, algorithm)` — stores a new cross-signature after verifying both devices belong to the same user
       - `get_device_trust_chain(user_id)` — returns all cross-signatures for a user, showing which devices trust which
       - `verify_device_trust(user_id, device_id)` — checks if a device has at least one cross-signature from another verified device
       - `revoke_device_trust(user_id, device_id)` — revokes all cross-signatures involving a device (used when a device is removed)

    3. Create the Ecto schema `CGraph.Crypto.E2EE.CrossSignature`:
       - `belongs_to :signer_device, CGraph.Crypto.E2EE.IdentityKey`
       - `belongs_to :signed_device, CGraph.Crypto.E2EE.IdentityKey`
       - `belongs_to :user, CGraph.Accounts.User`
       - Changeset validates required fields and same-user constraint

    4. Run migration: `mix ecto.migrate`

    IMPORTANT: Verify both device identity keys belong to the same user_id before storing a cross-signature. This is a critical security invariant.

  </action>

    - `mix compile --warnings-as-errors` passes (or just `mix compile` if the project has existing warnings)
    - `grep -rn "cross_sign\|CrossSignature" apps/backend/lib/cgraph/crypto/e2ee/` shows new module
    - `mix test test/cgraph/crypto/ --seed 0` passes (if tests exist)
  </verify>
  <done>

<task type="auto">
  <name>Task 2: Add key sync and cross-signing API endpoints</name>
  <files>
    apps/backend/lib/cgraph/crypto/e2ee/key_sync.ex
    apps/backend/lib/cgraph_web/controllers/api/v1/e2ee_controller.ex
    apps/backend/lib/cgraph_web/router/user_routes.ex
  </files>
  <action>
    1. Create `key_sync.ex` context module:
       - `create_sync_package(from_device_id, to_device_id, encrypted_key_material)` — stores encrypted key material for transfer between devices. The key material is encrypted client-side with the target device's public key — server never sees plaintext keys.
       - `get_pending_sync_packages(device_id)` — retrieves sync packages awaiting pickup by a device
       - `mark_sync_complete(package_id)` — marks a package as received

    2. Add 4 new endpoints to `e2ee_controller.ex`:

       a. `POST /api/v1/e2ee/devices/:device_id/cross-sign` → `cross_sign_device`
          - Body: `{ signature: base64, algorithm: string }`
          - Verifies caller owns an identity key, creates cross-signature for target device
          - Returns: `{ status: "verified", trust_chain: [...] }`

       b. `GET /api/v1/e2ee/devices/trust-chain` → `device_trust_chain`
          - Returns all cross-signatures for current user
          - Response: `{ devices: [{ device_id, identity_key, cross_signatures: [...], is_trusted }] }`

       c. `POST /api/v1/e2ee/devices/:device_id/sync` → `sync_keys`
          - Body: `{ encrypted_key_material: base64, target_device_id: uuid }`
          - Stores encrypted key package for target device
          - Returns: `{ package_id: uuid, status: "pending" }`

       d. `GET /api/v1/e2ee/devices/sync-packages` → `get_sync_packages`
          - Returns pending sync packages for current device
          - Response: `{ packages: [{ id, from_device_id, encrypted_key_material, created_at }] }`

    3. Register routes in `user_routes.ex`, under the existing E2EE scope:
       ```elixir
       post "/e2ee/devices/:device_id/cross-sign", E2EEController, :cross_sign_device
       get "/e2ee/devices/trust-chain", E2EEController, :device_trust_chain
       post "/e2ee/devices/:device_id/sync", E2EEController, :sync_keys
       get "/e2ee/devices/sync-packages", E2EEController, :get_sync_packages
       ```

    4. All endpoints require authentication (the existing `authenticated_routes` scope handles this).

    IMPORTANT: The server NEVER sees plaintext private keys. All key material is encrypted client-side before being sent to the sync endpoint. The server is a blind relay.

  </action>

    - `curl -s localhost:4000/api/v1/e2ee/devices/trust-chain -H "Authorization: Bearer $TOKEN"` returns JSON (when server running)
    - `mix test test/cgraph_web/controllers/ --seed 0` passes
  </verify>
  <done>
    4 new API endpoints for multi-device key sync: cross-sign, trust-chain, sync-keys, get-sync-packages. Backend handles encrypted key material as blind relay. Routes registered under authenticated E2EE scope.
  </done>
