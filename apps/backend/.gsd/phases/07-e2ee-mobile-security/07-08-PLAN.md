<task type="auto">
  <name>Task 1: Implement device sync protocol on web and mobile</name>
  <files>
    apps/web/src/lib/crypto/e2ee-store/device-sync.ts
    apps/mobile/src/lib/crypto/store/deviceSync.ts
  </files>
  <action>
    1. Create `device-sync.ts` (web) implementing the client-side sync protocol:

       a. `registerNewDevice()`:
          - Called during auto-bootstrap on a new device (from Plan 07-01's setupE2EE flow)
          - Generates new identity key + prekeys (already done by setupE2EE)
          - After registration, sends a "device_added" event via Phoenix user channel
          - Stores device_id in SecureStorage for future identification

       b. `requestCrossSign(existingDeviceId)`:
          - Shows notification to existing device via user channel event
          - Existing device receives notification → shows verification prompt
          - On approval: existing device creates signature of new device's identity key
          - Calls `POST /api/v1/e2ee/devices/:device_id/cross-sign` with the signature

       c. `syncKeyMaterial(targetDeviceId)`:
          — Existing device encrypts session state using target device's public key
          - Uses `@cgraph/crypto` KEM: encrypt with target device's identity key
          - Calls `POST /api/v1/e2ee/devices/:device_id/sync` with encrypted blob
          - New device polls `GET /api/v1/e2ee/devices/sync-packages` for encrypted material
          - New device decrypts with its private key and imports session state

       d. `getDeviceTrustChain()`:
          - Calls `GET /api/v1/e2ee/devices/trust-chain`
          - Returns list of user's devices with trust status

    2. Create `deviceSync.ts` (mobile) with the SAME protocol but using mobile-specific APIs:
       - Use `expo-secure-store` for key storage (instead of IndexedDB)
       - Use the same API endpoints
       - Same cross-signing flow

    3. Key sync flow:
       ```
       New device login → auto-bootstrap generates own keys → registers with server
         → Server notifies existing devices via user channel "device_added" event
         → Existing device shows "New device detected — verify?" prompt
         → User confirms → existing device cross-signs new device key
         → Existing device encrypts active session data for new device
         → New device polls and downloads encrypted sessions
         → New device decrypts and imports sessions
         → Both devices now have E2EE capability
       ```

    IMPORTANT: Per-device sessions (Signal pattern) — sender encrypts separately for each device. Server returns ALL device bundles when other users fetch bundles. The `get_prekey_bundle` endpoint already supports this.

  </action>

    - `pnpm --filter web lint && pnpm --filter mobile lint`
  </verify>
  <done>
    Device sync protocol implemented on both platforms: registration, cross-signing, encrypted key transfer, trust chain query. New devices can sync E2EE capability from existing devices.

<task type="auto">
  <name>Task 2: Device management UI and key change notifications</name>
  <files>
    apps/web/src/modules/chat/components/device-verification-dialog.tsx
    apps/mobile/src/screens/settings/device-management-screen.tsx

  <action>
    1. Create `device-verification-dialog.tsx` (web):
       - Triggered when user channel receives "device_added" event
       - Shows: "A new device is requesting to join your account"
       - Displays new device info (device_id, platform, last seen)
       - Actions: "Verify & Trust" (cross-signs) | "Reject" (ignores)
       - On "Verify & Trust": calls `requestCrossSign()` from device-sync.ts
       - After verification: optionally sync key material

    2. Create `device-management-screen.tsx` (mobile):
       - Navigate from Settings → "Linked Devices"
       - Shows list of user's devices from `getDeviceTrustChain()`
       - Each device shows: name/platform, trust status (verified/unverified), last active
       - "Remove Device" action → calls `DELETE /api/v1/e2ee/devices/:device_id` + `revokeDeviceTrust()`
       - "Verify" action for unverified devices → prompts cross-signing flow

    3. Add key change notification banner to both platforms:
       - When a contact's identity key changes (detected during session creation or from server notification), show a banner in the conversation:
         "⚠️ {Contact}'s security number has changed. Tap to verify."
       - Banner links to safety number screen/dialog (from Plan 07-06)
       - This integrates with the backend's existing `is_verified: false` reset when identity keys change

    4. On web: add "Linked Devices" section to the settings page (reuse the same API).

    IMPORTANT: The device verification dialog is critical for security — it prevents unauthorized devices from joining the account. The key change banner prevents MITM attacks after key rotation.
