# Plan 12-01: Roles & Permissions Hardening

## Context

The role and permission system has solid foundations:
- **Backend:** `role.ex` (151L) has 23-bit bitmask (view_channels → manage_emojis + administrator). `roles.ex` (181L) has CRUD + `calculate_permissions/2` that OR-combines member roles. `permission_overwrite.ex` (56L) has schema with allow/deny bitmask per role/member per channel.
- **Web:** Full role editor (648L across 6 files) + channel permissions panel (705L across 7 files)
- **Mobile:** Basic role screen (131L) — needs full permission toggle UI. No channel permissions screen exists.

**Critical gap:** `calculate_permissions/2` ignores channel overrides entirely — it only OR-combines role bitmasks. The `group_channel.ex` (288L) calls `Groups.can_view_channel?` and `can_send_messages?` but these don't factor in per-channel overrides from `channel_permission_overwrites` table.

## Tasks

### Task 1: Add manage_automod permission bit and effective permissions calculator

**Files:** `apps/backend/lib/cgraph/groups/role.ex`, `apps/backend/lib/cgraph/groups/roles.ex`

**Action:**
1. In `role.ex`, add `manage_automod: 1 <<< 23` to `@permissions` map. Update `@default_permissions` to NOT include it (moderator-only).
2. In `roles.ex`, add `calculate_effective_permissions/3` that takes `(member, group, channel)`:
   - Get base permissions from `calculate_permissions/2` (existing — OR of all role bitmasks)
   - If member is group owner, return all permissions (existing behavior)
   - If base has `administrator` bit set, return all permissions (bypass overrides)
   - Query `channel_permission_overwrites` for this channel
   - For each override matching the member's roles: apply allow bits (OR), then apply deny bits (AND NOT)
   - For any member-specific override: apply allow (OR), then deny (AND NOT) — member overrides trump role overrides
   - Return effective bitmask as integer
3. Add `has_effective_permission?/4` convenience: `(member, group, channel, permission_atom)` → boolean
4. Keep existing `calculate_permissions/2` for non-channel contexts (group-level permissions)

**Verify:** `mix compile --warnings-as-errors` passes. Unit test: member with `send_messages` denied via channel override → `has_effective_permission?` returns false for that channel.

**Done:** `calculate_effective_permissions/3` exists and correctly merges role + channel overrides with proper precedence.

### Task 2: Wire effective permissions into group_channel.ex

**Files:** `apps/backend/lib/cgraph_web/channels/group_channel.ex`, `apps/backend/lib/cgraph/groups.ex`

**Action:**
1. In `groups.ex`, add delegations:
   - `defdelegate calculate_effective_permissions(member, group, channel), to: Roles`
   - `defdelegate has_effective_permission?(member, group, channel, perm), to: Roles`
2. In `group_channel.ex`:
   - On `join/3`: after getting member, load the channel's group (already have channel.group_id). Store `channel` in socket assigns alongside `channel_id`.
   - Change `verify_channel_access/2` to call `Groups.has_effective_permission?(member, group, channel, :view_channels)` instead of `Groups.can_view_channel?(member, channel)`
   - Change `verify_send_permission/1` to accept `(member, group, channel)` and call `Groups.has_effective_permission?(member, group, channel, :send_messages)`
   - Update all callers of `verify_send_permission` in `handle_in("new_message", ...)` to pass group + channel from socket assigns
   - Update `verify_delete_permission` similarly — check `manage_messages` effective permission
   - Update `pin_message` handler — check `manage_messages` effective permission
3. Add `GET /api/v1/groups/:group_id/channels/:channel_id/permissions/:member_id` endpoint to `role_controller.ex` that returns the computed effective permissions as a map of `{permission_name: boolean}`. This lets clients display "effective permissions" view.

**Verify:** Start backend with `mix phx.server`. Request effective permissions endpoint. Verify a member with channel-override deny cannot send messages in that channel.

**Done:** `group_channel.ex` uses effective permissions for all authorization checks. REST endpoint returns computed permissions.
