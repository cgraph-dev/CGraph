  **Migration — `e2ee_sessions` table:**
  - `id` binary_id primary key
  - `user_id` references users NOT NULL
  - `peer_id` references users NOT NULL (the other participant)
  - `conversation_type` string NOT NULL ("dm" | "secret")
  - `conversation_id` binary_id NOT NULL
  - `device_id` string
  - `session_state` string default "active" (active, stale, terminated)
  - `current_ratchet_public_key` binary — latest ratchet public key from this user
  - `message_count` integer default 0
  - `last_key_rotation_at` utc_datetime_usec
  - `created_at` utc_datetime_usec
  - `updated_at` utc_datetime_usec

Index on `[user_id, peer_id, conversation_type]`. Unique index on
`[user_id, peer_id, conversation_id, device_id]`.

**Migration addition to `e2ee_identity_keys` table:**

- Add column `one_time_prekey_count` integer default 100
- Add column `prekey_low_watermark` integer default 25
- Add column `last_prekey_replenish_at` utc_datetime_usec

**`SecretSession` module:**

- `create_session(user_id, peer_id, conversation_id, conversation_type, opts)` — registers a new
  E2EE session. Opts: device_id.
- `update_ratchet_key(session_id, new_public_key)` — updates the current ratchet public key and
  increments message_count. Updates `last_key_rotation_at` if key changed.
- `get_session(user_id, peer_id, conversation_id)` — retrieves active session.
- `terminate_session(session_id)` — marks session as terminated.
- `check_key_rotation_needed?(session)` — returns true if message_count > 100 since last rotation.

**`RatchetState` module:**

- `verify_ratchet_advance(session, new_ratchet_header)` — validates that the ratchet header
  represents a valid advance (message number is monotonically increasing, no replay).
- `compute_safety_number(user_key, peer_key)` — computes 60-digit safety number from two identity
  keys (SHA-256 truncated, formatted as groups of 5). Same algorithm as Signal.
- `safety_number_qr_data(user_id, peer_id)` — returns data for QR code verification.

**Pre-key watermark monitoring:** Update `key_operations.ex` to add:

- `check_prekey_watermark(user_id)` — counts remaining one-time prekeys. If below watermark (25),
  broadcasts a `prekey_low` event to the user's channel so the client can upload more.
- `replenish_prekeys(user_id, new_prekeys)` — adds new one-time prekeys, updates count.
- Call `check_prekey_watermark` after every prekey consumption (when a new session is established).
