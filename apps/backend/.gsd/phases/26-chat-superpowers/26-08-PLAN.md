  **Disconnection Point #1: Auth → E2EE Bootstrap**

Update `E2EE` module to add bootstrap check:

- `check_bootstrap_status(user_id)` — returns:
  - `{:ok, :ready}` — identity key registered, ≥10 pre-keys available
  - `{:ok, :needs_prekeys, count_remaining}` — identity key exists but pre-keys running low
  - `{:error, :no_identity_key}` — user never registered E2EE keys
- This is called after authentication. The response tells the client what bootstrap steps to
  perform.

Update `KeyOperations`:

- `get_prekey_bundle(user_id, requesting_user_id)` — returns a pre-key bundle for initiating a
  session. Atomically claims one pre-key (marks as used). Returns:
  `%{identity_key, signed_prekey, one_time_prekey, pq_prekey}`.
- If no one-time pre-keys left, return bundle without one-time pre-key but trigger
  `prekey_exhaustion` notification via PubSub:
  `Phoenix.PubSub.broadcast(CGraph.PubSub, "user:#{user_id}", {:prekey_exhaustion, count: 0})`.
- `upload_prekeys(user_id, prekeys)` — batch upload new pre-keys (client sends after bootstrap or
  after exhaustion notification). Validate each prekey has: `{key_id, public_key}`. Limit: max 100
  pre-keys upload per batch. Return count of keys stored.
- `get_prekey_count(user_id)` — return count of unused pre-keys.

**Disconnection Point #2: Message → Encrypt → Channel**

Update `ConversationChannel.handle_in("new_message", ...)` to support encrypted message structure:

```elixir
# Client sends:
%{
  "content" => base64_ciphertext,     # encrypted message body
  "is_encrypted" => true,
  "ratchet_header" => %{              # opaque to server — forwarded as-is
    "dh_public" => base64_dh_key,     # current ratchet public key
    "pn" => previous_chain_length,
    "n" => message_number,
    "pq_kem_ct" => base64_kem_ct      # post-quantum KEM ciphertext (optional)
  },
  "session_id" => session_uuid,       # E2EE session identifier
  "reply_to_id" => optional_id
}
```

- Server MUST NOT parse or validate `content` when `is_encrypted: true` — it's opaque ciphertext.
- Server MUST forward `ratchet_header` as-is in broadcast to recipient.
- Server stores in DB: `content` (ciphertext), `is_encrypted: true`, `ratchet_header` (JSON map).
- Broadcast includes:
  `%{id, content, is_encrypted, ratchet_header, session_id, sender_id, inserted_at}`.
- For non-encrypted messages (`is_encrypted: false`), existing flow applies (XSS sanitization etc).

Update `after_join` message history to include `ratchet_header` in historical messages if present.

  **Update/Create `E2EEController` endpoints:**
  - `GET /api/v1/e2ee/bootstrap` — returns bootstrap status for current user.
    Response: `%{status: "ready"|"needs_prekeys"|"no_identity_key", prekey_count: N}`.
  - `POST /api/v1/e2ee/keys/identity` — register identity key (one-time, after account creation).
    Body: `%{identity_key: base64_pub, signed_prekey: %{key_id, public_key, signature}}`.
  - `POST /api/v1/e2ee/keys/prekeys` — upload batch of one-time pre-keys.
    Body: `%{prekeys: [%{key_id, public_key}, ...]}`. Max 100 per request.
  - `GET /api/v1/e2ee/keys/bundle/:user_id` — get pre-key bundle for initiating session with user.
    Response: `%{identity_key, signed_prekey, one_time_prekey, pq_prekey}`.
    Atomically claims one pre-key.
  - `GET /api/v1/e2ee/keys/count` — get current pre-key count for authenticated user.
  - `POST /api/v1/e2ee/keys/backup` — store encrypted key backup.
    Body: `%{device_id, encrypted_backup: base64_blob}`.
  - `GET /api/v1/e2ee/keys/backup/:device_id` — retrieve encrypted key backup.
  - `GET /api/v1/e2ee/sessions/:user_id/safety-number` — compute safety number for a session.
    Uses public identity keys of both users. Returns `%{safety_number: "60-digit-string", qr_data: base64}`.
