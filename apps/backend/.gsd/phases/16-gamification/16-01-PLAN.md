# Plan 16-01: XP Event Pipeline — Action Triggers, Daily Caps, Forum XP

## Context

GAME-01 and GAME-10 require XP to flow from real user actions. The backend has a complete `award_xp/4` function (Gamification facade, 455 LOC) with polynomial level curve, multipliers, XP transactions, leaderboard sync. But **nothing calls it** when users actually do things. This plan wires the event pipeline.

The existing `XpTransaction` schema tracks 17 source types: `:message`, `:forum_thread_created`, `:forum_post_created`, `:forum_upvote_received`, `:friend_added`, `:daily_login`, `:quest_reward`, etc. — currently used by `:daily_login` (streak claim), `:achievement` (AchievementSystem), `:quest` (QuestSystem), and event rewards (EventRewardDistributor).

**Key architecture decision:** Create a centralized `XpEventHandler` module that all contexts call through, rather than embedding gamification logic in each context. This keeps the messaging/forums/social contexts clean and makes XP rules configurable.

<objective>
Wire all user actions to XP/coin awards through a centralized event handler with daily caps and diminishing returns. Establish the forum-specific XP pipeline with per-board leaderboards.

Purpose: This is the foundational action→reward pipeline that all other gamification features depend on.
Output: XpEventHandler, XpConfig, DailyCap modules; integration hooks in messaging/forums/social; shared gamification types; XP toast components.
</objective>

<task type="auto">
  <name>Task 4: Create XP event handler</name>
  <files>apps/backend/lib/cgraph/gamification/xp_event_handler.ex</files>
  <action>
  Create `CGraph.Gamification.XpEventHandler` — the central action→reward pipeline:
  ```elixir
  def handle_action(user, action_type, opts \\ []) do
    with {:ok, config} <- validate_action(action_type),
         {:ok, effective_xp} <- DailyCap.check_and_increment(user.id, action_type, config.xp),
         {:ok, {updated_user, level_up}} <- Gamification.award_xp(user, effective_xp, to_string(action_type), opts) do
      
      # Auto-award companion coins (1 coin per 50 XP earned)
      coin_amount = config.coins + div(effective_xp, 50)
      if coin_amount > 0, do: Gamification.award_coins(updated_user, coin_amount, "xp_companion", opts)
      
      # Broadcast real-time XP event via PubSub
      broadcast_xp_event(updated_user, effective_xp, action_type, level_up)
      
      # Update quest progress for action type
      Gamification.update_quest_progress(user.id, action_type, 1)
      
      {:ok, %{user: updated_user, xp_awarded: effective_xp, level_up: level_up}}
    end
  end
  ```
  - `broadcast_xp_event/4` publishes to `Phoenix.PubSub` on topic `"gamification:#{user_id}"`
  - Include reference_type/reference_id in opts for traceability
  - Handle {:error, :daily_cap_reached} gracefully — still count for quest progress even if no XP
  - IMPORTANT: Skip processing for action types `:achievement`, `:quest`, and event reward sources — these already call award_xp directly via AchievementSystem/QuestSystem/EventRewardDistributor. XpEventHandler handles only user-initiated actions.
  - Async (Task.async) the broadcast + quest update to not block the calling context
  </action>
  <verify>cd apps/backend && mix compile --warnings-as-errors 2>&1 | grep -c "error" || echo "0 errors"</verify>
  <done>XpEventHandler compiles with handle_action/3 piping through caps, awards, broadcasts, and quest updates</done>
