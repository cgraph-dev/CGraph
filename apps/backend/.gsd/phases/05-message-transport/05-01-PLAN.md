# Plan 05-01: Web Real-Time Message Transport

**Phase:** 05-message-transport **Wave:** 1 (parallel with 05-02) **Depends on:** —
**Requirements:** MSG-01, MSG-06, MSG-18, MSG-19

## Goal

Wire the existing web socket events to the chat store and UI components so that 1:1 messages deliver
in real-time with delivery receipts, read receipts (with opt-out), and debounced typing indicators —
all running through the existing ConversationChannel.

## Context (from discovery)

**What already exists and works:**

- `ConversationChannel` in socket package (277 lines) — fully typed events for `new_message`,
  `message_updated`, `message_deleted`, `typing`, `read_receipt`, `reaction_added/removed`
- Web socket handler (`apps/web/src/lib/socket/conversationChannel.ts`, 203 lines) — wired for
  `new_message`, `message_updated`, `message_deleted`, `typing`, `reaction_added/removed`
- Chat Zustand store (`apps/web/src/modules/chat/store/`) — full CRUD, pagination, typing state,
  E2EE support, 500-message cap with LRU pruning
- `MessageStatusIndicator` component (99 lines) — renders 4 states (sending → sent → delivered →
  read) with animated checkmarks, **but not connected to real data**
- `TypingIndicator` component (74 lines) — animated dots, **already wired** to store
- `ConversationList` with online indicators, search, pins — **already implemented**
- `MessageList` with virtualized rendering via `@tanstack/react-virtual` — **already implemented**
- REST endpoints: all message CRUD, mark-read, typing — **all wired**
- Backend `ConversationChannel` (390 lines) — broadcasts `msg_delivered`, `message_read`, typing
  with 5s auto-clear, rate limiting (10 msg/10s)

**What's NOT wired (the gaps this plan closes):**

1. `read_receipt` socket event defined in package but NO handler in web socket layer
2. No `msg_delivered` / delivery confirmation handler in web socket layer
3. `Message` type in store has no `deliveryStatus` field — `MessageStatusIndicator` has nothing to
   render from
4. `markAsRead()` calls REST API but doesn't update per-message read status locally
5. `sendTyping()` has no client-side debounce — could flood the socket
6. `setTypingUser()` doesn't auto-clear after timeout — typing can appear "stuck"
7. Store lacks `readReceipts` tracking for showing who has read each message

## Tasks

### Task 1: Add delivery status to message types and store

**Files:**

- `packages/shared-types/src/models.ts`
- `apps/web/src/modules/chat/store/chatStore.types.ts`
- `apps/web/src/modules/chat/store/chatStore.messaging.ts`
- `apps/web/src/modules/chat/store/chatStore.message-ops.ts`

**Action:**

1. In `packages/shared-types/src/models.ts`, add `deliveryStatus` field to the `Message` interface:

   ```typescript
   deliveryStatus?: 'sending' | 'sent' | 'delivered' | 'read';
   ```

2. In `chatStore.types.ts`, add to `ChatState`:

   ```typescript
   readReceipts: Record<string, Record<string, string>>; // messageId → userId → readAt
   updateMessageStatus: (conversationId: string, messageId: string, status: Message['deliveryStatus']) => void;
   addReadReceipt: (conversationId: string, messageId: string, userId: string, readAt: string) => void;
   ```

3. In `chatStore.messaging.ts`, update `sendMessage` to:
   - Set `deliveryStatus: 'sending'` on the optimistic message added to the store
   - On successful API response or socket broadcast back, update to `'sent'`
   - This makes `MessageStatusIndicator` show the spinning clock → single checkmark transition

4. In `chatStore.message-ops.ts`, implement `updateMessageStatus` and `addReadReceipt`:
   - `updateMessageStatus` finds the message in the conversation's array and sets the field
   - `addReadReceipt` adds to the `readReceipts` record and also updates message status to `'read'`
     if the reader is the conversation partner

**Verify:** TypeScript compiles (`pnpm --filter web typecheck`). Store actions callable.

**Done:** Messages in the store carry `deliveryStatus`, and the store has actions to update it.

### Task 2: Wire read receipt + delivery receipt socket handlers

**Files:**

- `apps/web/src/lib/socket/conversationChannel.ts`
- `apps/web/src/modules/chat/store/chatStore.operations.ts`

**Action:**

1. In `conversationChannel.ts`, add handlers for:
   - `msg_delivered` event →
     `useChatStore.getState().updateMessageStatus(convId, msgId, 'delivered')`
   - `message_read` event → `useChatStore.getState().addReadReceipt(convId, msgId, userId, readAt)`
     AND `updateMessageStatus(convId, msgId, 'read')` (if reader is conversation partner)

   The backend `ConversationChannel` already broadcasts these events:
   - `msg_delivered` with `%{message_id, user_id, delivered_at}` (from L220–L230)
   - `message_read` with `%{message_id, user_id, read_at}` (from L238–L248)

2. In `chatStore.operations.ts`, update `markAsRead` to:
   - After the REST call succeeds, update local message statuses for all messages in that
     conversation up to the marked point to `'read'` (from the sender's perspective, this means
     their sent messages now show blue checkmarks because the recipient read them)

3. Add `msg_ack` push on receiving new messages from other users — when a message arrives via
   socket, auto-push `msg_ack` back to acknowledge delivery. This triggers the backend to broadcast
   `msg_delivered` to the sender. The backend already handles the `msg_ack` event (L215–L232).

**Verify:** In browser devtools Network/WS tab, sending a message shows `msg_delivered` event
received. Opening a conversation shows `message_read` events for the partner's messages.

**Done:** Read receipts and delivery receipts flow from backend → socket → store → UI is complete.

### Task 3: Connect MessageStatusIndicator to real data + typing refinements

**Files:**

- `apps/web/src/modules/chat/components/message-bubble/message-bubble.tsx`
- `apps/web/src/modules/chat/hooks/useConversationState.ts`
- `apps/web/src/lib/socket/conversationChannel.ts` (typing debounce)
- `apps/web/src/modules/chat/store/chatStore.operations.ts` (typing auto-clear)

**Action:**

1. In `message-bubble.tsx`, ensure `MessageStatusIndicator` receives the message's `deliveryStatus`:
   - The component already renders 4 states — it just needs the real `deliveryStatus` prop from the
     message object, which now exists after Task 1
   - Only show status indicator for messages sent by the current user
     (`message.senderId === currentUserId`)

2. In `useConversationState.ts` or the conversation page component, wire read receipt opt-out:
   - Check user settings for `showReadReceipts` preference
   - If opt-out enabled, don't push `mark_read` events (user's messages still show as "delivered" to
     senders, never "read")
   - Import settings from user preferences store

3. Add typing debounce in the socket layer:
   - In `conversationChannel.ts` or a helper, throttle `sendTyping(true)` to max once per 3 seconds
   - After 5 seconds of no keystroke, auto-send `sendTyping(false)`
   - Use a simple timer pattern — `lastTypingSentAt` + `typingTimeoutId`

4. Add typing auto-clear in the store:
   - In `chatStore.operations.ts`, when `setTypingUser(convId, userId, true)` is called, start a 6s
     timer. If no new typing event arrives for that user within 6s, auto-clear them (set to false).
   - This prevents "stuck typing" if the stop event is missed over the network.
   - Store the timer IDs in a Map outside the Zustand state (side-effect, not state).

**Verify:**

- Send a message → status shows "sending" → "sent" → "delivered" (when partner receives) → "read"
  (when partner opens conversation)
- Type rapidly → only one typing event per 3s in WS traffic
- Partner stops typing → indicator clears within 6s even if stop event dropped
- Disable read receipts in settings → partner's messages stay at "delivered" (no blue checkmarks)

**Done:** Full delivery lifecycle visible in UI. Typing indicators robust with debounce +
auto-clear.
