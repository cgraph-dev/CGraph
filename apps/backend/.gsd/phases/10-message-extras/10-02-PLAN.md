# Plan 10-02: Server-Side Link Preview Engine

## Context

The infrastructure is partially built:
- Message schema has `link_preview :map` field
- Web has `link-preview.tsx` component (82 lines) rendering title, description, image, favicon
- Mobile has `link-preview.tsx` component (149 lines) with similar rendering
- `message_params.ex` validates link_preview allowed keys
- `message_json.ex` serializes link_preview in responses

**The gap:** No server-side fetching exists. Currently clients must send `link_preview` data themselves. We need a backend service that:
1. Detects URLs in message content
2. Fetches OG metadata asynchronously via Oban worker
3. Caches results to avoid redundant fetches
4. Updates the message and broadcasts the preview to connected clients

## Existing Code to Modify

- `apps/backend/lib/cgraph/messaging/core_messages.ex` — trigger preview fetch after message creation
- `apps/backend/lib/cgraph_web/channels/conversation_channel.ex` — broadcast link_preview update
- `apps/backend/lib/cgraph/messaging/message.ex` — no changes needed (link_preview field exists)

## Tasks

### Backend

1. **Create `link_preview_cache` migration**
   - Table: `link_preview_cache`
   - Columns: `id`, `url_hash` (string, indexed — SHA256 of normalized URL), `url` (text), `title` (string), `description` (text), `image_url` (text), `favicon_url` (text), `site_name` (string), `type` (string), `fetched_at` (utc_datetime), `expires_at` (utc_datetime — TTL 7 days), timestamps
   - Unique index on `url_hash`

2. **Create `LinkPreviewCache` Ecto schema** (`link_preview_cache.ex`)
   - Fields matching migration
   - Changeset with URL validation
   - TTL default: 7 days from fetch

3. **Create `LinkPreviewService` module** (`link_preview_service.ex`)
   - `extract_urls(text)` — regex to find URLs in message content (http/https, handle markdown links)
   - `fetch_metadata(url)` — HTTP GET with:
     - 5-second timeout
     - Max 1MB response body (prevent abuse)
     - Follow up to 3 redirects
     - User-Agent header
     - Parse HTML for `<meta property="og:*">` and `<meta name="twitter:*">` tags
     - Fallback to `<title>` tag
     - Extract favicon from `<link rel="icon">`
   - `get_or_fetch(url)` — check cache first, if miss or expired → fetch and cache
   - `build_preview_map(metadata)` — returns map matching existing `link_preview` field format: `%{url, title, description, image, siteName, type, favicon}`

4. **Create `FetchLinkPreview` Oban worker** (`fetch_link_preview.ex`)
   - Queue: `:link_previews` (add to Oban config)
   - Max attempts: 2
   - Args: `%{message_id: uuid}`
   - Logic:
     1. Load message
     2. Extract URLs from content
     3. Take first URL only (avoid fetching dozens)
     4. Call `LinkPreviewService.get_or_fetch(url)`
     5. Update message's `link_preview` field
     6. Broadcast `"link_preview_updated"` event to conversation channel with `%{message_id, link_preview}`

5. **Trigger preview fetch on message creation**
   - In `core_messages.ex`, after successful message insert:
     - Check if content contains URLs (`LinkPreviewService.extract_urls/1`)
     - If yes, enqueue `FetchLinkPreview` worker with `%{message_id: message.id}`
   - Skip if message already has `link_preview` set (client sent it)

6. **Add Oban queue config**
   - Add `:link_previews` queue to Oban configuration (concurrency: 5)

7. **Handle channel broadcast**
   - In `conversation_channel.ex`, add handler or broadcast helper for `"link_preview_updated"` event
   - Or use `Phoenix.PubSub.broadcast` directly from worker

8. **Add cache cleanup worker** (optional but recommended)
   - Oban cron worker that deletes expired cache entries (older than 7 days)
   - Runs daily

### Web

9. **Handle real-time link_preview update**
   - In the chat store or socket handler, listen for `"link_preview_updated"` event
   - When received, update the message in the store with the new `link_preview` data
   - The existing `LinkPreview` component will render it automatically

10. **Remove client-side link_preview sending** (optional improvement)
    - If web is currently sending link_preview data when creating messages, this can be removed since server now handles it
    - Or keep as a fast-path (show immediately, server replaces later)

### Mobile

11. **Handle real-time link_preview update**
    - Same as web: listen for `"link_preview_updated"` channel event
    - Update message in mobile store
    - Existing `LinkPreview` component handles rendering

## Verification

- Send a message containing a URL → link preview appears within 2-3 seconds with title, description, image
- Send same URL again → preview appears instantly (cached)
- Send a URL to a page with no OG tags → falls back to `<title>` tag
- Send a message with no URLs → no preview fetch triggered
- Long/slow URL → times out after 5 seconds, no preview shown (graceful failure)
- Preview appears on both web and mobile via real-time update
