## Context

The X3DH (Extended Triple Diffie-Hellman) implementation computes only 3 of the 4 DH
operations. DH4 = ECDH(EK_A, OPK_B) provides additional forward secrecy by involving
the one-time prekey. The implementation note says the one-time prekey computation is
"not implemented" — this weakens the security guarantee.

The WatermelonDB bridge for offline messages stores minimal sender info (just ID),
which means offline-cached messages can't display sender names or avatars without a
network request to look up the user.

## Tasks

1. **Implement X3DH DH4**
   - Locate X3DH handshake function in `x3dh.ts`
   - After DH3 computation, check if `bundle.oneTimePrekey` exists
   - If yes: compute `DH4 = ECDH(ephemeralKey.privateKey, bundle.oneTimePrekey.publicKey)`
   - Update KDF input: `SK = KDF(DH1 || DH2 || DH3 || DH4)`
   - If no OPK: keep current 3-DH behavior, set `usedOneTimePrekey: false` flag
   - Server-side: mark one-time prekey as consumed after first use

2. **Handle one-time prekey lifecycle**
   - Receiving side: retain OPK private key until initial session message arrives
   - After receiving first message: securely zero-out OPK private key from storage
   - Replenish consumed OPKs by generating and uploading new ones

3. **Expand WatermelonDB message schema**
   - Add columns: `sender_display_name` (string), `sender_avatar_url` (string, nullable)
   - Update schema version + migration

4. **Cache sender profile in bridge**
   - When storing a message offline, include sender profile data
   - Read sender name/avatar from message metadata or fetch from profile cache
   - Display cached profile data in offline message list
   - Fallback: "Unknown User" with generic avatar if no profile cached
