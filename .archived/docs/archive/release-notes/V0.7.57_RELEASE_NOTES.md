## 🔒 Security Enhancements

### Critical Content Moderation System

The moderation system (`lib/cgraph/moderation.ex`) has been completely overhauled to handle critical
safety reports with industry-standard practices:

**NCMEC-Compatible Incident Reporting**

- Generates structured incident reports following CyberTipline format
- Includes preservation status, content hashes (placeholder), and submission readiness flags
- Audit logging for legal compliance with permanent retention policy

**Automatic Content Quarantine**

- Critical reports trigger immediate content hiding
- Supports messages, posts, comments, and profile content
- Visibility flag prevents content display while preserving for review

**On-Call Staff Alerting**

- Real-time PubSub broadcast to admin channels
- Background job queuing for email/SMS/webhook notifications
- Escalation deadline tracking (1-hour default)

**Warning System with Escalation**

- Warning severity calculated from violation category + user history
- 90-day lookback window for repeat offense tracking
- Escalation levels: minor → moderate → severe → critical → final_warning
- User notification with appeal deadline (7 days)

### Rate Limiting Overhaul

**Post Rate Limiting** (`lib/cgraph/forums.ex`)

```elixir
# Dynamic limits based on user trust level
:new_user    -> 0.5x multiplier (< 7 days, < 5 posts)
:regular     -> 1.0x multiplier (standard users)
:trusted     -> 2.0x multiplier (> 30 days, > 50 posts)
:veteran     -> 3.0x multiplier (> 180 days, > 200 posts)
```

**Comment Rate Limiting**

- Per-thread limits (10 comments/user/thread/hour) to prevent flooding
- Separate burst and sustained rate check pipeline
- Post ID context passed for thread-specific limiting

**Referral System Protection**

- Code regeneration: 3/day limit
- Code application: 5/hour limit
- Self-referral prevention with code owner lookup
- Fraud detection logging (IP, user-agent)

### Input Validation Layer

New validation utilities in `lib/cgraph_web/helpers/param_parser.ex`:

| Function                 | Purpose                               |
| ------------------------ | ------------------------------------- |
| `parse_referral_code/1`  | 6-12 alphanumeric, uppercased         |
| `validate_event_title/1` | 3-200 chars, prohibited content check |
| `validate_event_dates/2` | Start < end, not in past              |
| `validate_visibility/1`  | Whitelist validation                  |
| `validate_pagination/2`  | Configurable bounds                   |
| `sanitize_html/2`        | XSS vector stripping                  |

## 🏗️ Infrastructure Improvements

### Fly.io Configuration (`infrastructure/fly/fly.toml`)

| Setting      | Before | After  | Reason                    |
| ------------ | ------ | ------ | ------------------------- |
| Memory       | 512MB  | 1024MB | BEAM VM + WebSockets      |
| Kill timeout | 5s     | 30s    | Graceful connection drain |
| CPUs         | 1      | 2      | Scheduler performance     |
| Grace period | 10s    | 30s    | BEAM startup time         |
| Soft limit   | 800    | 1500   | Higher concurrency        |
| Hard limit   | 1000   | 2000   | Peak capacity             |
