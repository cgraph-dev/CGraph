## Context

The payout system (`lib/cgraph/creators/payout.ex` L29-30) performs a balance check then creates a
payout record without transaction isolation — concurrent requests can both see the full balance and
both succeed, causing double payouts.

The IAP controller (`lib/cgraph_web/controllers/iap_controller.ex`) processes Apple App Store and
Google Play notifications but does not verify the cryptographic signatures on incoming webhooks,
allowing attackers to forge free premium subscriptions.

The wallet authentication module (`lib/cgraph/accounts/wallet_authentication.ex`) parses SIWE
messages but does not validate the `chainId` field, allowing cross-chain replay attacks.

Audit logging infrastructure exists (`lib/cgraph/audit.ex` + entry_builder, persistence, query) but
not all security-sensitive paths emit entries.

## Tasks

1. **Fix payout race condition**
   - Wrap `request_payout/1` in `Repo.transaction`
   - Add `from(e in Earning, where: e.creator_id == ^creator_id, lock: "FOR UPDATE")` before balance
     check
   - Return `{:error, :insufficient_balance}` if balance < requested amount after lock
   - Write concurrent race test using `Task.async_stream`

2. **Verify Apple JWS signatures**
   - Add `JOSE` library JWS verification to IAP webhook handler
   - Fetch + cache Apple root CA certificates
   - Reject unverified payloads with 401
   - Add test with crafted valid and tampered JWS

3. **Verify Google RTDN auth tokens**
   - Validate Bearer token on Google Pub/Sub push messages
   - Use Google's public keys to verify JWT
   - Reject unverified messages with 401

4. **Validate SIWE chain_id**
   - Extract `chainId` from parsed SIWE message
   - Check against `Application.get_env(:cgraph, :allowed_chain_ids, [1])`
   - Return `{:error, :invalid_chain_id}` on mismatch

5. **Complete audit logging**
   - Add `Cgraph.Audit.log/1` calls to payout, IAP, and wallet auth paths
   - Include failed verification attempts as security audit events
