<objective>
Implement the QR code login protocol: scan from mobile to authenticate web sessions.

Purpose: AUTH-11 — users can scan a QR code from their already-authenticated mobile device to log
into the web app. This is a new protocol requiring: (1) backend session management in Redis, (2)
challenge-response HMAC verification, (3) WebSocket channel for real-time auth status, (4) web QR
display, (5) mobile QR scanner with confirmation. No existing QR login infrastructure exists — this
is fully new.

Protocol flow:

1. Web requests QR session → backend creates session in Redis (5min TTL)
2. Web displays QR code + subscribes to `qr_auth:{session_id}` channel
3. Mobile scans QR → decodes session_id + challenge
4. Mobile POSTs signed challenge (HMAC-SHA256) to `/api/v1/auth/qr-login`
5. Backend verifies → generates tokens → broadcasts on channel → web receives tokens → authenticated

Output: Full QR login protocol end-to-end (backend + web + mobile). </objective>

1. Create `apps/backend/lib/cgraph/auth/qr_login.ex` — QR login session management:
   - `create_session/0`: Generate UUID session_id + 32-byte random challenge. Store in Redis as JSON with key `qr_auth:{session_id}`, TTL 300s (5 min). Return `{:ok, %{session_id, challenge, expires_at, qr_payload}}` where qr_payload = Base.url_encode64(Jason.encode!(%{sid: session_id, ch: challenge, srv: CGraphWeb.Endpoint.url()}))
   - `get_session/1`: Fetch session by ID from Redis. Return `{:ok, session}` or `{:error, :not_found}` or `{:error, :expired}`
   - `verify_and_complete/3`: Takes session_id, user (the mobile user), and signature. Verifies: (a) session exists and is "pending", (b) HMAC-SHA256(challenge, user.id) matches signature. On success: generate access_token + refresh_token via existing auth token generation (check how AuthController.login generates tokens), mark session as "completed" in Redis with tokens, set short TTL (30s) for cleanup. Return `{:ok, %{access_token, refresh_token}}`
   - `delete_session/1`: Delete session from Redis
   - Use `CGraph.Redis.command/1` for Redis operations (existing pattern)
   - Use `:crypto.mac(:hmac, :sha256, key, data)` for HMAC verification
   - Rate limit: add a simple check — max 5 sessions per IP per 5 minutes using Redis INCR with TTL

2. Create `apps/backend/lib/cgraph_web/controllers/api/v1/qr_auth_controller.ex`:
   - `create_session/2` (POST, unauthenticated): Creates QR session via QrLogin.create_session(),
     returns JSON with session_id, qr_payload, expires_at
   - `approve_login/2` (POST, authenticated — requires current_user):
     - Params: `{ session_id, signature }`
     - Calls QrLogin.verify_and_complete(session_id, current_user, signature)
     - On success: broadcast `%{status: "authenticated", access_token: ..., refresh_token: ...}` to
       `qr_auth:{session_id}` via
       `CGraphWeb.Endpoint.broadcast("qr_auth:#{session_id}", "auth_complete", payload)`
     - Return 200 `%{status: "approved"}`
     - On failure: return 401/404 with error reason

3. Create `apps/backend/lib/cgraph_web/controllers/api/v1/qr_auth_json.ex` for JSON rendering.

4. Create `apps/backend/lib/cgraph_web/channels/qr_auth_channel.ex`:

     hasn't logged in yet)
   - On join: verify session exists in Redis (prevents random channel joins). If session not found,
     reject join.
   - No handle_in needed — this channel only receives broadcasts from the server
   - Handle `auth_complete` event — just broadcasts to subscribers (default Phoenix behavior)
