<objective>
Wire biometric authentication to app lock and E2EE key access on mobile.

Purpose: The biometric module (`biometrics.ts`) is fully implemented with Face ID / Touch ID /
Fingerprint support, timeout-based re-auth, and SecureStore preference persistence. The settings
screen already has a toggle. This plan wires biometrics to: (1) gate app access on foreground after
timeout, and (2) gate E2EE private key operations. Satisfies AUTH-06.

Output: Mobile app requires biometric auth after 5min inactivity. E2EE key loading is gated behind
biometric unlock. </objective>

<task type="auto">
  <name>Task 1: Wire biometric gate to app foreground lifecycle</name>
  <files>
    apps/mobile/src/app/_layout.tsx
  </files>
  <action>
    1. In the root layout (`_layout.tsx`), add an `AppState` listener for the `active` event (app returning to foreground):
       - Import `requireAuthenticationIfNeeded`, `isBiometricLockEnabled` from `../lib/biometrics`
       - On `AppState` change to `active`:
         a. Check `isBiometricLockEnabled()` — if false, skip
         b. Call `requireAuthenticationIfNeeded('Unlock CGraph')` — this uses the 5-minute timeout already built into `biometrics.ts`
         c. If biometric fails: show a full-screen lock overlay (opaque, covering all content) with a "Tap to unlock" button that retries biometric auth
         d. If biometric succeeds: dismiss overlay, resume normal app

    2. Create a `BiometricLockOverlay` component inline in the layout (or as a small separate component if the layout is already complex):
       - Full-screen View with app logo, "CGraph is locked" text, and "Unlock" button
       - On button press → call `authenticateWithBiometrics('Unlock CGraph')` again
       - On success → hide overlay via state toggle

    3. The overlay MUST render above all navigators/screens — use React state in the root layout, not in a nested screen.

    4. Handle edge case: if device has no biometric hardware or biometric not enrolled, skip the gate entirely (the biometric toggle in settings should be disabled in this case — `biometrics.ts` already handles this via `getBiometricStatus()`).

    IMPORTANT: Do NOT use `expo-local-authentication` directly — always go through the `biometrics.ts` wrapper which handles all edge cases.

  </action>

    - `pnpm --filter mobile typecheck` passes
    - `grep -n "requireAuthenticationIfNeeded\|AppState" apps/mobile/src/app/_layout.tsx` shows biometric gate
    - `pnpm --filter mobile lint` passes
  </verify>
  <done>
    App foreground triggers biometric check when enabled. Lock overlay blocks UI until authenticated. 5-minute timeout from `biometrics.ts` prevents excessive prompts.
  </done>

       - Import `requireAuthenticationIfNeeded`, `isBiometricLockEnabled` from `../../biometrics`
       - In the `loadIdentityKeyPair` or equivalent key-loading function:
         a. Check `isBiometricLockEnabled()`
         b. If enabled: call `requireAuthenticationIfNeeded('Access encryption keys')` before reading from SecureStore
         c. If biometric fails: throw a specific error (e.g., `BiometricAuthRequired`) that callers can catch and show appropriate UI
         d. If disabled: proceed normally (no biometric prompt)
       - This gates ALL crypto operations behind biometric since they all need the identity key

    2. In `account-screen.tsx`, verify the existing biometric toggle:
       - Confirm it calls `setBiometricLockEnabled(enabled)` on toggle
       - Confirm it checks `getBiometricStatus()` on mount and disables the toggle if hardware not available
       - If the toggle doesn't exist yet in the actual rendered UI (research says it does), add it:
         a. Toggle label: "Biometric Lock" with subtitle showing biometric type (Face ID / Fingerprint)
         b. Below: status text ("Face ID enabled" / "Touch ID enabled" / "Not available")
       - Add a biometric verification prompt BEFORE enabling: the user must authenticate to turn ON biometric lock (prevents unauthorized enabling)

    3. Handle the edge case where queued encrypted messages arrive while app is locked:
       - Store incoming encrypted messages in the message queue
       - Decrypt them AFTER biometric unlock on app foreground
       - This integrates with the existing WatermelonDB bridge pattern from Phase 6

    IMPORTANT: Keys are already stored with `WHEN_UNLOCKED_THIS_DEVICE_ONLY` in SecureStore. The biometric gate adds an additional layer on top of the OS-level protection.
