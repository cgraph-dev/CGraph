decrypt pipeline in the message receive handler and renders a lock icon on encrypted messages.
Completes the end-to-end encryption loop on web for E2EE-01.

Output: Incoming encrypted messages are decrypted and displayed. Per-message lock icon shows E2EE
status. Success criteria #1 satisfied. </objective>

  <name>Task 1: Add decrypt-on-receive to message handler</name>
  <files>
    apps/web/src/modules/chat/store/chatStore.message-ops.ts
    apps/web/src/modules/chat/store/chatStore.messaging.ts
  </files>
  <action>
    1. In `chatStore.message-ops.ts`, modify the `addMessage` function (the handler called when Phoenix Channel delivers a `new_message` event):
       - After receiving a message, check `message.is_encrypted` (or `message.isEncrypted` depending on the normalized field name)
       - If encrypted AND the message is NOT from the current user (don't decrypt our own sent messages — we already have the plaintext):
         a. Get the e2eeStore instance (import `useE2EEStore` or access it via a module-level reference)
         b. Call `e2eeStore.getState().decryptMessage(message.senderId, { ciphertext: message.content, header: message.header, protocol_version: message.protocol_version, nonce: message.nonce, ephemeralPublicKey: message.ephemeral_public_key })`
         c. Replace `message.content` with the decrypted plaintext
         d. Set `message.isEncrypted = true` (flag for UI lock icon)
         e. If decryption fails: set `message.content = "⚠️ Unable to decrypt this message"` and `message.decryptionFailed = true`

    2. If the `Message` type in `chatStore.types.ts` doesn't have `isEncrypted` and `decryptionFailed` fields, add them:
       - `isEncrypted?: boolean`
       - `decryptionFailed?: boolean`
       - `protocolVersion?: string`

    3. Handle the session initialization race: if no session exists for the sender when trying to decrypt, the session manager should auto-create a responding session using the message's key exchange data. The existing `pqxdhRespond` flow in the session manager should handle this. Verify this works for the PQXDH case.

    4. For messages received while E2EE is still initializing (auto-bootstrap hasn't completed), queue them for later decryption. After `initialize()` completes, process the queue.

    CRITICAL BEHAVIOR: Never display ciphertext to the user. Either show decrypted plaintext or show an error placeholder. Never fall back to showing raw encrypted bytes.

  </action>

    - `pnpm --filter web test -- --testPathPattern="chatStore" --passWithNoTests` passes
  </verify>
  <done>
    Incoming encrypted messages are decrypted via e2eeStore before display. Handles both PQXDH and X3DH sessions. Decrypt failures show error placeholder. Messages queued during E2EE init.
  </done>
