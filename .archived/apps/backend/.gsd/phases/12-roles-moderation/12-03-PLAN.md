# Plan 12-03: Automod Enforcement Pipeline

## Context

**Existing:** Backend has automod CRUD (`automod_rule.ex` 40L schema, `automod.ex` 67L context,
`automod_controller.ex` 126L REST). Web has `automod-settings.tsx` (272L). Schema supports:
word_filter, link_filter, spam_detection, caps_filter with actions: delete, warn, mute,
flag_for_review.

**Critical gap:** Zero runtime enforcement exists. `group_channel.ex` has **no automod integration**
— messages are broadcast directly without any rule checking. The entire automod system is CRUD-only
with no actual filtering.

**Depends on 12-01:** Uses `manage_automod` permission bit for authorization checks. Uses effective
permissions for mute action enforcement.

## Tasks

### Task 1: Build automod enforcement engine

**Files:** `apps/backend/lib/cgraph/groups/automod/enforcement.ex` (NEW),
`apps/backend/lib/cgraph/groups/automod.ex`

**Action:**

1. Create `automod/enforcement.ex`:

   ```elixir
   defmodule CGraph.Groups.Automod.Enforcement do
     @moduledoc "Runtime message checking against automod rules."

     alias CGraph.Groups.{Automod, AutomodRule}

     @type check_result :: :ok | {:blocked, AutomodRule.t()}

     @doc "Check message content against all enabled rules for the group."
     @spec check_message(binary(), binary(), binary()) :: check_result
     def check_message(group_id, content, sender_id)
   end
   ```

2. Implementation:
   - `check_message/3`: Fetch enabled rules for group (cache with ETS or process dictionary for
     performance — rules don't change often). Run each rule checker in order:
     - `check_word_filter/2`: Split `pattern` by comma → check if any word exists in lowercased
       content (support regex patterns)
     - `check_link_filter/2`: Extract URLs from content with regex → check against blocked domain
       patterns
     - `check_spam_detection/3`: Track recent messages per user per channel in ETS/Agent — detect
       repeated content (>3 identical messages in 60s) or message flood beyond channel rate limit
     - `check_caps_filter/2`: Calculate uppercase ratio — block if >70% caps and message length >10
   - Return `{:blocked, rule}` on first matching rule, or `:ok` if all pass

3. `execute_action/4` — `(rule, message_or_content, member, socket)`:
   - `:delete` → Don't broadcast. Reply to sender with
     `{:error, %{reason: "automod_blocked", rule: rule.name}}`
   - `:warn` → Broadcast message but also push private "automod_warning" event to sender with rule
     name
   - `:mute` → Create temporary send_messages restriction for member (add to a `group_mutes` table
     or use existing member field). Duration: 5 minutes for first offense, escalating
   - `:flag_for_review` → Broadcast message (visible) but create a report via
     `Moderation.Reports.create_report/2` with `reporter_id: nil` (system),
     `target_type: "message"`, `category: "automod_flag"`,
     `description: "Flagged by automod rule: #{rule.name}"`

4. Add to `automod.ex`:
   - `seed_default_rules/1` — for a group_id, create 2 default rules:
     - Word filter: common slurs/spam terms (enabled, action: delete)
     - Link filter: known phishing domains (enabled, action: flag_for_review)
   - `get_enabled_rules/1` — cached query for active rules per group

**Verify:** `mix compile` passes. Unit test: `check_message/3` with word_filter containing "spam"
blocks "this is spam content".

**Done:** Automod enforcement engine exists with all 4 rule types and all 4 action types.

### Task 2: Integrate automod into group_channel.ex message flow

**Files:** `apps/backend/lib/cgraph_web/channels/group_channel.ex`,
`apps/backend/lib/cgraph/groups.ex`

**Action:**

1. In `groups.ex`, add delegation:
   `defdelegate check_automod(group_id, content, sender_id), to: Automod.Enforcement, as: :check_message`

2. In `group_channel.ex`, modify `handle_in("new_message", ...)`:
   - After `check_rate_limit` and `verify_send_permission` pass, BEFORE creating the message:
   - Call `Groups.check_automod(socket.assigns.group_id, content, user.id)`
   - Pattern match result:
     - `:ok` → continue to `handle_message_creation` (existing flow)
     - `{:blocked, rule}` → call `Automod.Enforcement.execute_action(rule, content, member, socket)`
       and return the appropriate reply (error for delete, warning for warn, etc.)
   - For `:mute` action: after muting, also update the socket assigns so subsequent messages in this
     connection are blocked without DB round-trip

3. Add automod bypass for administrators: if member has `administrator` effective permission, skip
   automod check entirely.

4. Wire `seed_default_rules/1` into group creation:
   - In `Groups.create_group/2` (or wherever groups are created), after successful insert, call
     `Automod.seed_default_rules(group.id)`

**Verify:** Send a message containing a blocked word in a group channel → message not broadcast,
sender receives automod error. Send as administrator → message goes through.

**Done:** All group channel messages pass through automod. Administrators bypass. Default rules
seeded on group creation.
