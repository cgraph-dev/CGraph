# Plan 12-04: Custom Emoji Permissions + Group E2EE Key Distribution

## Context

### Custom Emoji (GROUP-07)

**Existing:** Full CRUD backend (`custom_emoji.ex` 36L schema, `emojis.ex` 52L context,
`custom_emoji_controller.ex` 445L with upload/search/categories). Web emoji tab (231L) + upload
modal (163L). Mobile custom emoji screen (287L).

**Gap:** No permission enforcement — anyone can upload/delete. `manage_emojis` bit (22) exists in
role.ex but `emojis.ex` context doesn't check it. No animated emoji validation.

### Group E2EE (E2EE-02)

**Existing:** Comprehensive 1:1 E2EE: backend (1,466L across 6 files — key registration, sync,
generation, operations, cross-signing). Web (100+ files referencing e2ee — session manager,
verification). Mobile (1,600L — e2ee.ts 890L, e2eeStore 557L, context 153L).

**Gap:** Zero group E2EE code. No Sender Key protocol. No `group_sessions` table. No key
distribution. This is architecturally complex but builds on the existing E2EE foundation.

**Depends on 12-01:** Uses `manage_emojis` effective permission from the hardened permission system.

## Tasks

### Task 1: Enforce emoji upload permissions and validate animated emoji

**Files:** `apps/backend/lib/cgraph/groups/emojis.ex`,
`apps/backend/lib/cgraph_web/controllers/api/v1/custom_emoji_controller.ex`

**Action:**

1. In `custom_emoji_controller.ex`:
   - In `create/2` (upload): Before processing upload, verify the calling user's membership +
     `manage_emojis` permission via
     `Groups.has_effective_permission?(member, group, nil, :manage_emojis)` (nil channel =
     group-level permission)
   - In `delete/2`: Same permission check
   - In `update/2`: Same permission check
   - Read actions (index, show, search, popular, categories) remain public — any member can view/use
     emoji
   - Return `{:error, :forbidden}` → 403 response if permission denied

2. In `emojis.ex`:
   - Add `create_group_emoji/4` variant that accepts `member` and checks permission before
     delegating to existing creation
   - Add animated emoji validation in creation path:
     - Check file extension: `.gif` or `.apng` when `is_animated: true`
     - Check file size: max 256KB for animated, 128KB for static
     - Validate image dimensions: max 128x128 pixels
     - Return `{:error, :invalid_file}` with descriptive message on validation failure

3. Update response: include `can_manage` boolean in emoji list responses based on requesting user's
   permission, so clients can conditionally show upload/delete buttons.

**Verify:** Upload emoji as member without manage_emojis → 403. Upload as admin → success. Upload
oversized animated emoji → validation error.

**Done:** Emoji uploads gated by permission. Animated emoji validated.

### Task 2: Design and implement Sender Key group E2EE schema

**Files:** `apps/backend/lib/cgraph/crypto/e2ee/group_session.ex` (NEW),
`apps/backend/priv/repo/migrations/TIMESTAMP_create_group_e2ee_sessions.exs` (NEW)

**Action:**

1. Create migration `create_group_e2ee_sessions`:

   ```sql
   create table(:group_e2ee_sessions, primary_key: false) do
     add :id, :binary_id, primary_key: true
     add :group_id, references(:groups, type: :binary_id, on_delete: :delete_all), null: false
     add :user_id, references(:users, type: :binary_id, on_delete: :delete_all), null: false
     add :device_id, :string, null: false
     add :sender_key_id, :string, null: false           -- unique key identifier
     add :public_sender_key, :binary, null: false        -- the public part of sender key
     add :chain_key_index, :integer, default: 0          -- message index for ratcheting
     add :is_active, :boolean, default: true
     timestamps()
   end

   create unique_index(:group_e2ee_sessions, [:group_id, :user_id, :device_id])
   create index(:group_e2ee_sessions, [:group_id, :is_active])
   ```

   Also create `group_sender_key_distributions`:

   ```sql
   create table(:group_sender_key_distributions, primary_key: false) do
     add :id, :binary_id, primary_key: true
     add :session_id, references(:group_e2ee_sessions, type: :binary_id, on_delete: :delete_all)
     add :recipient_user_id, references(:users, type: :binary_id), null: false
     add :recipient_device_id, :string, null: false
     add :encrypted_sender_key, :binary, null: false     -- sender key encrypted for recipient
     add :distributed_at, :utc_datetime_usec
     timestamps()
   end

   create index(:group_sender_key_distributions, [:recipient_user_id, :recipient_device_id])
   ```

2. Create `group_session.ex` Ecto schema matching the migration.

**Verify:** `mix ecto.migrate` runs cleanly. Schema compiles.

**Done:** Group E2EE tables exist for Sender Key storage and distribution.

### Task 3: Build group key distribution backend

**Files:** `apps/backend/lib/cgraph/crypto/e2ee/group_key_distribution.ex` (NEW),
`apps/backend/lib/cgraph/crypto/e2ee.ex`

**Action:**

1. Create `group_key_distribution.ex`:
   - `register_sender_key/4` — `(group_id, user_id, device_id, public_sender_key)` → Insert into
     `group_e2ee_sessions`. Called by client after generating local Sender Key.
   - `distribute_key/5` —
     `(session_id, recipient_user_id, recipient_device_id, encrypted_sender_key)` → Insert into
     `group_sender_key_distributions`. Called by sender to share their key with each recipient
     (encrypted per-recipient using their identity key).
   - `get_session_keys/2` — `(group_id, user_id)` → Returns all active sender key distributions FOR
     this user (i.e., other members' keys encrypted for them). Used on channel join to load keys.
   - `get_group_members_keys/1` — `(group_id)` → Returns list of `{user_id, device_id, session_id}`
     for all active sessions. Used by new members to know who they need to encrypt their sender key
     for.
   - `rotate_keys/2` — `(group_id, reason)` → Mark all current sessions as `is_active: false`. Each
     client will need to generate a new sender key and re-distribute. Called when a member leaves/is
     kicked.
   - `invalidate_user_sessions/3` — `(group_id, user_id, device_id)` → Deactivate specific user's
     sessions (on leave/kick). Don't rotate all — only invalidate the leaving user's keys.
   - `cleanup_stale_sessions/1` — `(group_id)` → Remove sessions for users no longer in the group.

2. Add to `e2ee.ex` (main facade):
   - `defdelegate register_group_sender_key(...)`, etc. — expose through main E2EE module

3. Add WebSocket events in `group_channel.ex`:
   - On `join/3` (after_join): push `"e2ee_session_keys"` event with the joining member's received
     key distributions
   - Handle `"register_sender_key"` — client registers their sender key for this group
   - Handle `"distribute_sender_key"` — client distributes their encrypted sender key to a specific
     device
   - Handle `"request_key_distribution"` — new member requests keys from existing members (triggers
     server push to existing members asking them to distribute)
   - On member leave: broadcast `"key_rotation_required"` event to remaining members

**Verify:** `mix compile` passes. Register a sender key → stored in DB. Query session keys → returns
distributions.

**Done:** Full Sender Key protocol backend: register, distribute, rotate, invalidate.

### Task 4: Build client-side group E2EE (web + mobile)

**Files:** `apps/web/src/lib/crypto/group-e2ee.ts` (NEW),
`apps/web/src/modules/groups/store/groupE2eeStore.ts` (NEW),
`apps/mobile/src/lib/crypto/group-e2ee.ts` (NEW)

**Action:**

1. Create `apps/web/src/lib/crypto/group-e2ee.ts`:
   - `generateSenderKey()` → Returns `{senderKeyId, publicKey, privateKey}` using Web Crypto API
     (ECDH P-256 or X25519)
   - `encryptGroupMessage(content, senderKey)` → AES-256-GCM encrypt with ratcheting chain key.
     Returns `{ciphertext, senderKeyId, chainIndex}`
   - `decryptGroupMessage(ciphertext, senderKeyId, chainIndex, senderPublicKey)` → Decrypt using
     stored sender key
   - `encryptSenderKeyForRecipient(senderKey, recipientIdentityPublicKey)` → Encrypt sender key
     using recipient's identity key (ECDH + AES wrap)
   - `decryptReceivedSenderKey(encryptedKey, ownIdentityPrivateKey)` → Decrypt a sender key shared
     by another member
   - Chain ratcheting: increment chain_key_index on each encrypt, derive next chain key via HKDF

2. Create `apps/web/src/modules/groups/store/groupE2eeStore.ts`:
   - Zustand store: `senderKeys: Map<groupId, {key, id, chainIndex}>`,
     `peerKeys: Map<senderKeyId, {publicKey, chainIndex}>`
   - Actions: `initGroupE2EE(groupId)`, `handleSessionKeys(keys)`,
     `encryptMessage(groupId, content)`, `decryptMessage(senderKeyId, ciphertext, chainIndex)`
   - Persist sender keys to IndexedDB (existing pattern from 1:1 E2EE store)

3. Create `apps/mobile/src/lib/crypto/group-e2ee.ts`:
   - Mirror web implementation using existing mobile crypto utilities (expo-crypto or native module)
   - Same API surface: `generateSenderKey`, `encryptGroupMessage`, `decryptGroupMessage`,
     `encryptSenderKeyForRecipient`, `decryptReceivedSenderKey`
   - Integrate with existing `e2eeStore.ts` or create dedicated `groupE2eeStore.ts`

4. Wire into message flow (light integration — full E2EE toggle is infrastructure):
   - In group channel send path: if group has E2EE enabled, call `encryptGroupMessage` before
     sending via WebSocket. Include `encrypted_content`, `sender_key_id`, `chain_index` in message
     payload.
   - In receive path: if message has `encrypted_content`, call `decryptGroupMessage` using stored
     peer sender key.
   - Handle `e2ee_session_keys` event on join: store received keys.
   - Handle `key_rotation_required`: generate new sender key, register, distribute to all peers.

**Verify:** Generate sender key → encrypt message → decrypt with stored key → content matches. Key
distribution flow: member A registers key, distributes to member B, member B decrypts it
successfully.

**Done:** Group E2EE Sender Key protocol implemented on web + mobile. Messages can be
encrypted/decrypted client-side.
