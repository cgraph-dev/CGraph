<objective>
Wire the existing TokenManager (rotation + theft detection + family tracking) into the AuthController
refresh endpoint, replacing the simpler Guardian.refresh_tokens call. Validate with TDD.

Purpose: AUTH-14 requires that concurrent 401s trigger exactly one refresh and that token reuse is
detected as theft. The frontend already has a mutex/queue in `httpClient.ts` preventing concurrent
refresh requests from the client side. The backend needs to use `TokenManager` (which has rotation,
family tracking, and reuse detection) instead of the naive `Guardian.refresh_tokens` (which has none
of these). This is the highest-risk item in Phase 2.

Output: Secure token refresh with rotation, theft detection, and test coverage. </objective>

  <behavior>
    Token refresh behavior (input → expected output):

    1. Valid refresh token → {:ok, %{access_token: new_access, refresh_token: new_refresh}}
       - Old refresh token is invalidated (rotation)
       - New tokens have correct TTLs (access: 15min, refresh: 7d)
       - Response includes both tokens + user data

    2. Expired refresh token → {:error, :token_expired}
       - Returns 401 with clear error message

    3. Reused (already-rotated) refresh token → {:error, :token_reuse_detected}
       - Entire token family is revoked (theft detection)
       - All sessions for that device/family are invalidated
       - Returns 401 with "session_revoked" error

    4. Invalid/malformed refresh token → {:error, :invalid_token}
       - Returns 401 with "invalid_token" error

    5. Concurrent refresh with same token → Only first succeeds
       - Second request gets {:error, :token_already_used} or the same new tokens
       - Frontend mutex prevents this, but backend should be safe regardless

    6. Max sessions enforcement → oldest session evicted when limit (10) reached

  </behavior>
  <implementation>
    1. In `AuthController.refresh/2`:
       - Replace `CGraph.Guardian.refresh_tokens(refresh_token)` with
         `CGraph.Auth.TokenManager.refresh(user, refresh_token)` (or adapt the call signature)
       - Handle new error cases: `:token_reuse_detected`, `:token_family_revoked`
       - Map TokenManager response to the existing JSON response format

    2. In `Guardian.ex`:
       - Keep `refresh_tokens/1` for backward compatibility but mark as `@deprecated`
       - Or remove if nothing else calls it (check with grep)

    3. Verify TokenManager.Store is properly configured:
       - Redis connection available (check config)
       - ETS table initialized on application start
       - If Redis is not available in dev, ensure ETS-only fallback works

    4. Test file structure:
       ```elixir
       defmodule CGraph.Auth.TokenRefreshTest do
         use CGraph.DataCase, async: false  # Redis/ETS state

         describe "refresh/2" do
           test "valid refresh token returns new token pair"
           test "rotated token cannot be reused"
           test "reused token revokes entire family"
           test "expired token returns error"
           test "invalid token returns error"
         end

         describe "concurrent refresh" do
           test "concurrent refresh with same token - only first succeeds"
         end
       end
       ```

  </implementation>
