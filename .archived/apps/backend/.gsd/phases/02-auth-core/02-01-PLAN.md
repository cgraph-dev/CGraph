  <name>Task 2: Fix Sessions.ex field name mismatches</name>
  <files>apps/backend/lib/cgraph/accounts/sessions.ex</files>
  <action>
    The `Sessions` module builds changesets using `:token` and `:device` fields, but the `Session`
    schema (session.ex) defines `:token_hash`, `:device_name`, and `:device_type`. Ecto's `cast/4`
    silently ignores fields not in the schema, causing sessions to be created with NULL values.

    Fix the changeset construction in `Sessions.create_session/2` (and any other functions that
    build session changesets):
    1. Map `:token` ã `:token_hash` (hash the token with SHA256 before storing, matching the
       `SessionManagement` pattern: `:crypto.hash(:sha256, token) |> Base.encode64()`)
    2. Map `:device` ã split into `:device_name` and `:device_type` (parse from user_agent string
       if available, or accept structured input)
    3. Ensure `:ip_address`, `:user_agent`, `:location` are passed through from conn metadata
    4. Set `:expires_at` based on refresh token TTL (7 days default from Guardian config)

    Also check `revoke_session/1` and `touch_session/1` ã ensure they query by `:token_hash` not `:token`.

  </action>
  <verify>
    Run `mix compile --warnings-as-errors` ã no warnings.
    Run `mix test test/cgraph/accounts/sessions_test.exs` if exists.
    Manually verify: `grep -n "token_hash\|device_name\|device_type" apps/backend/lib/cgraph/accounts/sessions.ex`
    ã should show correct field names throughout.
  </verify>
  <done>

    device_name, device_type, and expires_at values. Queries use token_hash for lookups.
  </done>
