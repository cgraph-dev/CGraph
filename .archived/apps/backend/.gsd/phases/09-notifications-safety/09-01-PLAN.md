# Plan 09-01: Per-Conversation & Per-Channel Notification Preferences

## Context

The backend already has global notification toggles (`notify_messages`, `notify_mentions`, etc.) in
`user_settings.ex`, and per-group notification settings via
`PATCH /groups/:id/members/me/notifications`. However, there's **no per-conversation or per-channel
muting**. Users need granular control: mute a noisy conversation without silencing everything.

Forum subscriptions already have per-target notification modes (`instant/digest/none`) via
`subscription_params.ex` — we extend this pattern to conversations and channels.

## Existing Code to Modify

- `apps/backend/lib/cgraph/notifications/delivery.ex` — add per-target preference check
- `apps/backend/lib/cgraph/accounts/settings.ex` → `should_notify?/2` — add per-target override
- `apps/web/src/modules/chat/components/conversation-header.tsx` — add mute menu item
- `apps/mobile/src/screens/messages/conversation-screen.tsx` — add mute action
- `packages/shared-types/src/notifications.ts` — create new file with shared types

## Tasks

### Backend

1. **Create `notification_preferences` migration**
   - Table: `notification_preferences`
   - Columns: `id`, `user_id` (FK), `target_type` (string: "conversation", "channel", "group"),
     `target_id` (uuid), `mode` (string: "all", "mentions_only", "none"), `muted_until`
     (utc_datetime, nullable — for timed mutes), timestamps
   - Unique index on `(user_id, target_type, target_id)`

2. **Create `NotificationPreference` Ecto schema** (`notification_preference.ex`)
   - Belongs to User
   - Changeset with mode validation ("all", "mentions_only", "none")
   - `muted_until` nullable — nil means permanently muted until explicitly unmuted

3. **Create `Notifications.Preferences` context module** (`preferences.ex`)
   - `get_preference(user_id, target_type, target_id)` — returns preference or default (:all)
   - `set_preference(user_id, target_type, target_id, attrs)` — upsert via `on_conflict`
   - `mute(user_id, target_type, target_id, opts \\ [])` — set mode to "none", optional `until`
   - `unmute(user_id, target_type, target_id)` — delete preference (restores default)
   - `should_deliver?(user_id, target_type, target_id, notification_type)` — checks preference +
     expiry
   - `list_muted(user_id)` — returns all muted targets
   - `bulk_get_preferences(user_id, targets)` — for conversation list muted state

4. **Create `NotificationPreferenceController`**
   - `GET /api/v1/notification-preferences/:target_type/:target_id` — get preference
   - `PUT /api/v1/notification-preferences/:target_type/:target_id` — upsert preference (body:
     `{mode, muted_until}`)
   - `DELETE /api/v1/notification-preferences/:target_type/:target_id` — delete (unmute)
   - `GET /api/v1/notification-preferences` — list all preferences for current user
   - Add routes in `user_routes.ex`

5. **Integrate into delivery pipeline**
   - In `delivery.ex`, before calling push/web notification send, check
     `Preferences.should_deliver?`
   - If mode is "mentions_only", check if user is mentioned in the message
   - If mode is "none", skip delivery entirely
   - If `muted_until` has passed, treat as unmuted

6. **Write tests**
   - Context tests for `Preferences`: CRUD, should_deliver? logic, expired muted_until
   - Controller tests: auth, validation, upsert behavior

### Shared Types

7. **Create `packages/shared-types/src/notifications.ts`**

   ```typescript
   export type NotificationMode = 'all' | 'mentions_only' | 'none';

   export interface NotificationPreference {
     targetType: 'conversation' | 'channel' | 'group';
     targetId: string;
     mode: NotificationMode;
     mutedUntil?: string; // ISO datetime
   }
   ```

### Web

8. **Create `conversation-notification-settings.tsx`**
   - Dialog/popover triggered from conversation header "more" menu
   - Options: "All messages", "Mentions only", "Mute" (with optional duration picker: 1h, 8h, 24h, 1
     week, forever)
   - Calls `PUT /api/v1/notification-preferences/conversation/:id`
   - Shows current state

9. **Add mute indicator to conversation list**
   - In conversation list item, show muted icon (bell-off) when preference is "none"
   - Fetch muted state via `GET /api/v1/notification-preferences` on list mount

10. **Add mute to conversation header menu**
    - Add "Mute conversation" / "Unmute conversation" to existing header dropdown

### Mobile

11. **Create `conversation-notification-settings.tsx` (mobile)**
    - Bottom sheet triggered from conversation header action menu
    - Same options as web: All / Mentions only / Mute (with duration)
    - Same API calls

12. **Add mute indicator to conversation list**
    - Show muted icon in conversation list cells
    - Fetch preferences on list mount

## Verification

- Mute a conversation → send message → no push notification received
- Set "mentions only" → send regular message → no notification; send @mention → notification
  received
- Set timed mute (1h) → after 1h passes → notifications resume
- Unmute → notifications resume immediately
- Check both web and mobile
- Verify global settings still work alongside per-target settings
