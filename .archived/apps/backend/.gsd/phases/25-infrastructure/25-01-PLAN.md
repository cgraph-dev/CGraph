## Context

**PgBouncer**: Config exists at `pgbouncer/pgbouncer.ini` and `pgbouncer/start.sh` but is not
deployed. Under load, Phoenix will exhaust PostgreSQL's direct connection limit.

**MeiliSearch**: The search module has MeiliSearch integration code but production search falls back
to PostgreSQL ILIKE. MeiliSearch needs to be activated and kept in sync.

**Elixir version**: Dockerfile uses Elixir 1.17.3 but local dev uses 1.19.4. This can cause subtle
behavioral differences (compiler warnings, OTP version differences).

**Auth latency**: p95 is 383ms against a 300ms SLO. Likely caused by bcrypt computation rounds or an
unoptimized DB query in the auth flow.

**CRDT**: Group state uses CRDTs for distributed sync but has no compaction — state grows unbounded
as users make changes.

## Tasks

1. **Deploy PgBouncer**
   - Add PgBouncer as Fly.io process in `fly.toml`
   - Update `DATABASE_URL` to route through PgBouncer
   - Test connection pooling under load
   - Verify pgbouncer.ini settings (pool_mode=transaction, max_client_conn, default_pool_size)

2. **Activate MeiliSearch**
   - Configure MeiliSearch connection in `runtime.exs`
   - Ensure search module routes queries to MeiliSearch when available
   - Create/verify Oban job for index sync
   - Test fallback to PostgreSQL on MeiliSearch downtime

3. **Run load tests**
   - Review/create k6 scripts for: login, message send, forum browse, search
   - Run against staging
   - Record baseline metrics
   - Identify bottlenecks from results

4. **Fix auth p95 latency**
   - Profile auth flow: bcrypt verification time, DB query time, token generation
   - Common fixes: reduce bcrypt rounds for dev (keep 12 for prod), add DB index on email/username
     lookup, cache hot auth data in ETS
   - Re-run load test to verify < 300ms p95

5. **Align Elixir version**
   - Update Dockerfile: `FROM elixir:1.19.4` (or whatever is latest stable)
   - Update `.tool-versions` if needed
   - Run full test suite against new version
   - Verify no deprecation warnings

6. **Implement CRDT compaction**
   - Create Oban worker that periodically compacts CRDT state
   - Compaction: merge all concurrent updates into single state, reset causal context
   - Schedule: daily for active groups, weekly for inactive
   - Test: compacted state equals uncompacted state (just smaller)
